## Long-term

- [ ] **Re-evaluate LaunchPadBooster issue #6 (synchronized mod settings) once it lands**: https://github.com/StationeersLaunchPad/LaunchPadBooster/issues/6 proposes a wrapper around `BepInEx.Configuration.ConfigEntry` that holds both the locally-configured value and the currently-in-use value, with the host pushing the in-use value on connect so clients adopt it without rewriting their local config. Today our mods solve server-authoritative values case by case: `PowerTransmitterPlus` ships `DistanceConfigMessage` and `BeamVisualConfigMessage` with manual `SettingChanged` broadcasts and a `PlayerConnected` postfix to rebroadcast on join. When issue #6 ships, audit every `ConfigEntry` across `Mods/` and `Plans/` that is marked `(Server-authoritative)` in its description and decide per-entry whether to migrate. Entries whose values are safe to hot-swap at runtime (beam color, distance cost factor, pulse width, etc.) are good migration candidates; entries that gate Harmony patch application at boot (for example `PowerTransmitterPlus.EnableAutoAim` when that lands) are NOT, because the wrapper does not solve "patch state can only be set at process start" and those still need the `IJoinValidator` handshake approach. Produce a short migration plan per mod, keep the handshake path for boot-time gates, migrate the hot-swap values, delete the bespoke sync messages once the wrapper covers the same ground.
