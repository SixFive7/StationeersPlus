# KeypadMod Fix Playtests

Implemented changes awaiting in-game confirmation. An agent records a playtest here, not in `TODO.md`, when it has changed code whose behavior can only be confirmed by running the game: single-player, a hosted multiplayer session, or the dedicated server under `DedicatedServer/`. This keeps `TODO.md` a list of work still to do, and gives one place to check whether a change already has a pending test before adding another.

Rules:
- Add an entry when code is implemented but its in-game behavior is unconfirmed. Write down everything a tester needs: what changed (commit if there is one), single-player vs multiplayer / dedicated-server, the save or world to set up, the exact in-game steps, what to watch (InspectorPlus request files, specific log lines, on-screen behavior, IC10 reads), and the expected result. Point at any staged `.work/<date>-<slug>/` request files or playbook.
- Check first. Before adding, scan the entries below so a change already covered by a pending test is not duplicated; extend the existing entry instead.
- Remove an entry when one of these happens: a run confirms it works; a run shows it broken (then add a fresh `TODO.md` item for the fix, or keep working on it now); or the player says the playtest is done. Entries are plain bullets, not checkboxes: like `TODO.md`, finished items are removed, not ticked off. Outcomes live in git history.

Prerequisite for both tests: BepInEx + StationeersLaunchPad + KeypadMod (by WIKUS) + KeypadMod Fix all installed and enabled on each machine under test. Deploy `KeypadModFix.dll` + its `About/` folder into the local mods folder (and into `DedicatedServer/` for the server test). On load, the BepInEx log should show `KeypadModFix v1.0.0 loaded; patches deferred until all mods are loaded.` followed by `KeypadMod Fix active: 2 of 2 patch(es) applied to keypadmod.Keypad.`. If it instead logs `KeypadMod ... is not installed` or `1 of 2`, the deploy or KeypadMod is wrong; stop and report that line.

- **Single-player: fix 1 (keypress crash) and host screen path.** Load any single-player save (or a fresh world). Build or creative-spawn an `ItemKitKeypad`, place and power it. Step 1: press several number buttons on the keypad. Expected: it responds with no error; the BepInEx / Player.log shows NO `MissingMethodException: ... UniTask.Delay`. Before this mod, the first keypress threw that exception and froze the keypad. Step 2: open the keypad screen (the Set action), type a number, submit. Expected: the keypad's logic value updates to that number (single-player uses the host path, which already worked, so this is a no-regression check). Optional verification: after submitting, copy `.work/2026-05-31-keypadmodfix/inspector-requests/screen_after.json` into `<StationeersInstall>\BepInEx\inspector\requests\`, then read the newest `...\inspector\snapshots\snapshot_*.json` and confirm the `keypadmod.Keypad` entry's `Setting` equals the number you typed and `_interactionLocked` is back to false.

- **Dedicated server + multiplayer client: fix 2 (screen input never applied).** This is the bug a dedicated-server player reported ("shows 0, can't set a value"). Deploy the four mods into `DedicatedServer/` (see `DedicatedServer/CLAUDE.md`), start the server, and connect as a client. Place and power a keypad. As the CLIENT, open the keypad screen, type a number, submit. Expected: the value applies and the screen shows the new number. Before this fix, a client's submit did nothing (the field kept its old value, usually 0) because the client sent the logic message with no `LogicType`, so the server dropped it. Verification with InspectorPlus on the SERVER: drop `.work/2026-05-31-keypadmodfix/inspector-requests/screen_before.json` into the server's `BepInEx\inspector\requests\` before the client submits, then `screen_after.json` right after; read the two newest snapshots and confirm the `keypadmod.Keypad` `Setting` changed from its old value to the number the client typed. Also press number buttons as the client and confirm the SERVER log shows no `MissingMethodException` (fix 1 runs server-side too).
