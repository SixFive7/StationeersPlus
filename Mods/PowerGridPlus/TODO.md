### What was ported from Re-Volt, and what was dropped

- **Ported** (renamed, MIT notice retained in file headers): the power-tick rewrite (`RevoltTick` -> `PowerGridPlus.Power.PowerGridTick`), `CableNetworkPatches` (swap the `PowerTick` per network), `PowerTickPatches` (route `Initialise`/`CalculateState`/`ApplyState` to the new tick + reverse-patch `CacheState` and `CheckForRecursiveProviders`), `DevicePatches` (`Device.AssessPower` re-implementation), `StationaryBatteryPatches`, `BatteryLogicPatches`, `TransformerExploitPatches`, `TransformerLogicPatches`, `AreaPowerControlPatches`.
- **Dropped**: all circuit-breaker / smart-breaker / heavy-breaker classes, the Load Center, the item kits and their `electronics.xml` recipes, `PrefabPatcher`, `ThingPatches`, the visual/animator components, `CircuitBreakerSaveData`, the `IBreaker`/`IPatchable`/`ISwitchgearComponent`/`ILoadCenter` interfaces, the `MultiConstructor`/`SingleConstructor` subclasses, and the `Heavy Breaker Maximum Trip Setting` / `Enable Custom Objects` config entries. This drops the LibConstruct/StationeersMods dependency -- the mod is pure BepInEx + StationeersLaunchPad.
- **`PowerGridTick` simplifications vs `RevoltTick`**: the `PowerClass` enum, the `PowerStates[]` array, `GetPowerState`, `ClassifyDevice`, and the load-center wiring were all removed (they only existed to support the Load Center). `DevicePatches.AssessPower` was simplified to match (no `GetPowerState` gate). `RevoltTick`'s `float.IsNormal(...)` NaN guard was replaced with `float.IsNaN(x) || float.IsInfinity(x)` (`float.IsNormal` is .NET Core 3.0+ / not in .NET Framework 4.7.2; the replacement is behaviourally equivalent for "is this a usable finite watt count"). `MathF.Min` (Re-Volt's `StationaryBatteryPatches`) was replaced with `UnityEngine.Mathf.Min` for the same portability reason.
- **APC patch -- fixed an apparent Re-Volt bug**: Re-Volt's `AreaPowerControllerPatches` declares two `[HarmonyPrefix]` patches both targeting `AreaPowerControl.GetUsedPower`, the first of which takes a `powerUsed` parameter that only matches `AreaPowerControl.UsePower`. Power Grid Plus targets `UsePower` for that one (which is what the body implements). If the in-game load reveals the original was right after all, revert.
- **Risk -- reverse patches**: `PowerTickPatches.CacheState`/`CheckForRecursiveProviders` (both confirmed to exist as private methods on `PowerTick` in the decompile), `DevicePatches.SetPower` (`Device.SetPower`, assumed from Re-Volt), `StationaryBatteryPatches.GetIsOperable` (`Battery.get_IsOperable`, assumed from Re-Volt). If any target doesn't resolve, `harmony.PatchAll()` throws and the try/catch in `Plugin` logs it as fatal -- check the BepInEx log first thing when testing.
- **Risk -- patch signatures assumed from Re-Volt source** (not all re-verified against the 0.2.6228.27061 decompile): `Device.AssessPower(CableNetwork, bool isOn)`, the `Transformer`/`Battery`/`AreaPowerControl` `GetGeneratedPower`/`GetUsedPower`/`ReceivePower`/`CanLogicRead`/`GetLogicValue` overrides, the `____powerProvided` private-field name. Re-Volt 1.4.0 targets a similar game version, so these should hold, but the load log is the first place to look.

### NEW-3 -- three-tier transmission voltages

- **Cable-tier incompatibility (the headline) -- working, two enforcement paths:**
  - `VoltageTierPatches.Cable_OnRegistered_Postfix`: after a cable registers and merges, if its network now holds more than one `CableType`, `VoltageTier.ResolveMixedTierNetwork` burns one lowest-tier cable (preferring the cable that just registered, since that's the bridge the player placed). The re-flood splits the network; the next rebuild re-checks; it converges over a few ticks.
  - `PowerGridTick.Initialize_New` detects a mixed-tier network on rebuild and requests the same resolution (covers loaded saves with pre-existing junctions, and anything `OnRegistered` missed). Guarded by a per-tick `_tierResolutionPending` flag so it doesn't queue a second burn before the first split takes effect.
  - **Deviation -- coarse "boundary" heuristic**: only the just-registered cable is treated as the preferred victim; otherwise *some* lowest-tier cable in the network is burned, not necessarily the one geometrically at the junction. In a line like `heavy-heavy-normal-heavy-heavy` this is fine (one burn). In `normal-normal-heavy` it may burn two normals instead of one. Converges either way. Refining this needs the cable grid-adjacency API.
  - **Sharp edge -- retroactive enforcement on load**: with `EnableVoltageTiers` defaulting on, loading *any* existing save that has a normal cable touching a heavy cable (legal in vanilla, possibly common in real bases) will start burning cables on first power tick. The plan (D11) specifies exactly this ("a one-time boundary-cable cascade on first tick"), but it is destructive on first install. Worth reconsidering: on first load, log a warning listing the mixed networks and give the player a grace period, or require a config opt-in for retroactive enforcement. Left as the plan specifies for now.
- **Generators-on-heavy (D13) -- working at the tick level:** `PowerGridTick.CalculateState_New` zeroes a device's generated power if `EnableVoltageTiers` + `Generators Require Heavy Cable` are on, the device is a generator, and the network is not all-heavy. So a solar panel / wind turbine / fuel generator / RTG / Stirling engine / gas-pipe turbine on a normal or super-heavy network produces nothing.
  - **Deviation -- no build-time rejection for generators yet.** A player can still *place* a generator on a normal cable; it just won't generate. The plan wanted a `Device.CanConstruct` postfix rejecting it with a message; that needs the device-to-adjacent-cable lookup at preview time, which wasn't worked out tonight. Add it. (The tick-level enforcement is the real teeth either way.)
  - The "is this a generator?" discriminator (`VoltageTier.IsGenerator`) is a type whitelist: `IPowerGenerator` (covers `SolarPanel`, `WindTurbineGenerator`, `LargeWindTurbineGenerator`) plus `TurbineGenerator`, `StirlingEngine`, `PowerGeneratorSlot` (covers `SolidFuelGenerator`), `PowerGeneratorPipe` (covers `GasFuelGenerator`), `RadioscopicThermalGenerator`. Wireless `PowerReceiver` is **not** in the list (it relays, doesn't generate) -- revisit whether a wireless relay's large output should also be heavy-only.
- **Build-time cable-tier rejection -- best effort only:** `VoltageTierPatches.Cable_CanConstruct_Postfix` tries `CableNetwork.ConnectedNetworks(__instance)` at placement-preview time and rejects with "Cannot connect: different transmission voltage -- use a transformer." if a connected network already carries a different tier. Wrapped in try/catch -- if the preview cable isn't gridded yet (or the API shape differs), it does nothing and the reactive burn-on-join handles it post-placement. Needs in-game verification that `ConnectedNetworks` works on a preview cable.
- **Device-tier policy beyond generators -- NOT implemented.** The plan (D8) also wanted: ordinary devices restricted to normal cable, stationary batteries restricted to heavy, super-heavy as a backbone with no devices at all, and a configurable high-draw-machine whitelist (Carbon Sequester / Advanced Furnace / Arc Furnace) allowed on heavy. None of that placement gating exists yet -- only the cable-tier-incompat rule and the generator rule. Without it, you *can* hang anything off heavy/super-heavy cable; it just isn't lore-correct.
- **Transformer tier mapping (D9/D10) -- NOT implemented.** The plan wanted Small = heavy<->normal only, Medium = heavy<->heavy flow-limiter only, Large = super-heavy<->heavy only, all bidirectional, re-implementing `Transformer.GetGeneratedPower`/`GetUsedPower`/`ReceivePower` symmetrically. Tonight only Re-Volt's transformer *exploit fix* is ported (one-directional, vanilla flow direction). The cable-tier rule already makes a transformer the only thing that can bridge tiers (its input and output sit on separate networks, so no burn fires) -- but any transformer prefab can bridge any tier pair, and transformers are still one-directional. Distinguishing the prefabs needs their prefab names; the bidirectional rewrite is a separate chunk.
- **In-game tier names / rejection message**: the plan (D12) wants "Normal / Heavy / Super-Heavy cable" and the message "Cannot connect: different transmission voltage -- use a transformer". The rejection message string is in the code; no in-game name changes were made.

## Considered features (not yet decided)

These would all sit on top of the v1 design. None needs a new prefab, so they stay within the pure-patch stance, but each expands the feature surface and needs an explicit go/no-go.

- [ ] **Cable-heat dissipation (from Deadly Electricity).** Convert a small fraction of a cable network's throughput into heat dumped into the cable run's local atmosphere each tick -- "wires get warm under load". Config-toggled, default off (flavour, not a fix). Idea source: the [Deadly Electricity](https://steamcommunity.com/sharedfiles/filedetails/?id=3575959825) mod's "cable efficiency loss -> heat with network size" mechanic. Open questions: how much heat (a small % of `Consumed`?), where to inject it (the cable's `WorldGrid` atmosphere, like `WorldManager.Spark` does on rupture), whether it scales with network size or throughput, and whether it interacts badly with the Re-Volt-derived proportional-sharing tick. Decompile the relevant `AtmosphericEventInstance` / `MoleEnergy` injection path before committing.
- [ ] **Absorb Omni Transmitter Settings.** Fold the [Omni Transmitter Settings](https://steamcommunity.com/sharedfiles/filedetails/?id=3643534844) mod's behaviour into Power Grid Plus: configurable max transmit range, distance-based charge-power falloff, and a minimum-power-per-battery floor for the vanilla Omni (Microwave) Transmitter (`PowerTransmitterOmni`). Adds a `Server - Wireless` config section. Expands the mod from "wired power sim" to "wired sim + omni-transmitter tuning" (a deliberate scope widening that overrides the earlier "wireless tuning out of scope"). Still a pure patch (patch `PowerTransmitterOmni`, no new prefab). Check the Omni Transmitter Settings source/license first -- if available and permissive, adapt with attribution; otherwise reimplement from scratch. Don't conflict with this repo's separate Power Transmitter Plus mod, which handles the Microwave Power Transmitter dish (a different device).
- [ ] **Craftable RTG (nuclear recipe).** Add a survival crafting recipe for the RTG (`RadioscopicThermalGenerator`; the prefab exists and currently only deconstructs as a "Creative RTG"). Use vanilla nuclear materials (uranium-line ores / refined products -- check what's craftable without a third-party "Processed Uranium" dependency). Recipe-only (no new prefab), so it fits the pure-patch stance; mechanically the same kind of `RecipeData` / gamedata patch as NEW-2's cost bump. Precedents: [CraftableRTG](https://steamcommunity.com/sharedfiles/filedetails/?id=3026880031), [Survival RTG](https://steamcommunity.com/sharedfiles/filedetails/?id=2464235697), [Nuclear Recipes](https://steamcommunity.com/sharedfiles/filedetails/?id=3004097828). Under D13 a crafted RTG goes on `heavy` cable like every generator. Config-toggled. Decide which machine prints it, the recipe, and whether it stays "Creative RTG" on deconstruction or returns the kit.
