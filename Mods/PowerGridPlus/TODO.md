## Known vanilla bugs worked around

- **`CableNetwork.DataDeviceList.get` checks the wrong dirty flag.** Confirmed still present in game 0.2.6228.27061 (decompile lines 253519-253529): the getter refreshes on `PowerDeviceListDirty`, not `DataDeviceListDirty`, so a `DirtyDataDeviceList()`-only call leaves `_dataDeviceList` stale on the next read. PowerGridPlus already works around this by calling `DirtyPowerAndDataDeviceLists()` everywhere (see `TransformerPassthroughLogicPatches` / `InheritedLogicablePassthroughLogicPatches`) instead of the data-only dirty. ## Decisions and deviations from the 2026-05-12 / 05-13 build sessions

This is the record of choices made and corners cut while building the mod (the user asked for it to be carried as far as possible, with every deviation logged here for review). The mod **compiles** (`dotnet build -c Release` is green) and **loads cleanly on the dedicated server** (all Harmony patches apply, no errors) but has **not had its runtime behaviour play-tested** -- the actual power sim, cable burns, voltage-tier enforcement, and the battery/transformer/APC fixes are unverified.

### Round 3 (2026-05-13): build-time-device-rejection -> reactive cable-burn, plus burn-reason tooltips

After Round 2 the only NEW-3 gap was "build-time cursor reject for misplaced devices". On reflection the user preferred to not gate device placement at all and instead let the cable that the misplaced device is on burn -- symmetric with the existing cable-tier rule. Done:

- **`PowerGridTick.CalculateState_New` no longer zeroes a misplaced device's power.** Instead it records the first misplaced device of the tick (`_misplacedDeviceForBurn`); `ApplyState_New` burns the cable adjacent to it (via `device.PowerCables` filtered by `CableNetwork`). The device itself is never destroyed.
- **Both tier-burns moved to `ApplyState_New`, gated on `powerFlow > 0`.** An idle/off network destroys nothing, even if mixed-tier or with a misplaced device. As soon as actual power flows (Required > 0 *and* Potential > 0), the burn fires. Mixed-tier burns take precedence over device-tier burns (root cause first). The existing `_tierResolutionPending` flag still scopes one burn per tick instance.
- **`Cable_OnRegistered_Postfix` removed.** Cable registration triggers `DirtyPowerAndDataDeviceLists` -> `IsDirty = true` -> `Initialize_New` re-detects mixed-tier on the next tick -> `ApplyState_New` burns when power flows. The dirty-flag path is the same as for loaded saves; one code path covers both.
- **`Cable_CanConstruct_Postfix` extended** with a second pass over `Cable.ConnectedDevices()` -- placing a cable next to an existing device on a tier it doesn't accept is now rejected at the cursor with "Wrong voltage -- {device} doesn't accept {tier} cable". (Cable-to-cable cursor reject is unchanged.) Confirmed valid on the cursor ghost per `Research/Patterns/CursorAdjacencyLookup.md`.
- **Burn-reason tooltips.** Each call site that invokes `Cable.Break()` registers a reason string in `BurnReasonRegistry` keyed by the dying cable's `LocalGrid`. A postfix on `CableRuptured.OnRegistered(Cell)` consumes that entry and attaches it to the wreckage via a `ConditionalWeakTable<Thing, string>` sidecar. A postfix on `Thing.GetPassiveTooltip` filtered to `__instance is CableRuptured` appends "Burned: <reason>" to the tooltip's `Extended` field. Four reason categories so far: overload (vanilla-style sustained throughput), wrong voltage (mixed-tier bridge), wrong voltage (adjacent misplaced device), and the un-tagged fallback (some future game path we didn't author). Threading: `BurnReasonRegistry._pendingByCell` is a `ConcurrentDictionary<Grid3, string>` -- multiple power-tick worker threads can register concurrently safely. See `Research/GameClasses/Cable.md` (Wreckage section, 2026-05-13) for the curated game-internals.

Net result: build-time gating is cable-side only; device placement is never blocked. The "you put it on the wrong cable" feedback is the cable burning, with a tooltip on the wreckage explaining why. The runtime checks are bounded by network size and only do work on dirty rebuilds (cable tier) + the existing per-device tick loop (device tier) -- no extra dirty-flag plumbing was needed.

### Round 2 (2026-05-13): the previously-stubbed/deferred items, now resolved

After the first pass the user asked for everything stubbed/deferred/needing-a-call to be finished. Their answers (and what was done):

- **Retroactive enforcement on load**: keep the D11 design -- the first power tick after load burns the lowest-tier boundary cable of each mixed-tier network, splitting it. No grace period, no opt-out toggle. (Already implemented; unchanged. Still a sharp edge on first install -- see the NEW-3 note below -- but it's what was asked for.)
- **APCs (and transformers) in the tier scheme**: no per-device tier restriction and no per-prefab tier mapping -- they bridge whatever they're wired to. The single universal rule is the cable one (two cable tiers in one network -> burn the smaller); a transformer's / APC's input and output are always separate networks, so they never trigger it. This **supersedes the old D9/D10** (Small=heavy<->normal, Medium=heavy<->heavy, Large=superHeavy<->heavy, bidirectional rewrite) -- none of that was implemented and it isn't going to be; the universal rule does the job. Implemented: `Transformer`, `AreaPowerControl`, `WirelessPower`/`PowerReceiver`/`PowerTransmitter`, `PowerTransmitterOmni` are all tier-exempt in `VoltageTier.IsAllowedOnTier`.
- **Heavy-cable device list (D8)**: a built-in whitelist of the community-known high-draw machines -- `CarbonSequester`, `FurnaceBase` (covers `Furnace` + `AdvancedFurnace`), `ArcFurnace`, `Centrifuge`, `Recycler`, `IceCrusher`, `HydraulicPipeBender`, `DeepMiner` (covers `CombustionDeepMiner`) -- plus a `Extra Heavy-Cable Devices` config string (comma-separated PrefabNames) for modded machines. These may use heavy *or* normal cable, not super-heavy. (I did not exhaustively verify each one's draw exceeds 5 kW against the game data this session; `CarbonSequester`/`AdvancedFurnace`/`ArcFurnace` are verified from the decompile, the rest are the well-known set. If one shouldn't be there, drop it from `VoltageTier.IsHighDrawMachine`.)
- **Ordinary device on heavy/super-heavy = hard reject + no power if it slips through**: the "no power if it slips through" half is implemented (`PowerGridTick.CalculateState_New` zeroes a device's used and generated power when it's on a tier `VoltageTier.IsAllowedOnTier` rejects). The "hard reject at build time" half is **NOT done** -- it needs a device-to-adjacent-cable lookup at placement-preview time (`Device.CanConstruct`), which wasn't worked out. So today: an ordinary device placed on heavy/super-heavy cable places fine but receives no power; same for a generator/battery on the wrong tier. The build-time rejection (red cursor + message) for devices is the one remaining NEW-3 gap. (Cable-on-cable build-time rejection IS in, best-effort, via `Cable.CanConstruct` + `CableNetwork.ConnectedNetworks`.)
- Also dropped: the `Generators Require Heavy Cable` sub-toggle. There's now one toggle, `Enable Voltage Tiers`, for the whole NEW-3 feature. Added: the `Extra Heavy-Cable Devices` config string. Both under `Server - Voltage Tiers`.

### NEW-3 -- three-tier transmission voltages

- **Cable-tier incompatibility (the headline) -- working, two enforcement paths:**
  - `VoltageTierPatches.Cable_OnRegistered_Postfix`: after a cable registers and merges, if its network now holds more than one `CableType`, `VoltageTier.ResolveMixedTierNetwork` burns one lowest-tier cable (preferring the cable that just registered, since that's the bridge the player placed). The re-flood splits the network; the next rebuild re-checks; it converges over a few ticks.
  - `PowerGridTick.Initialize_New` detects a mixed-tier network on rebuild and requests the same resolution (covers loaded saves with pre-existing junctions, and anything `OnRegistered` missed). Guarded by a per-tick `_tierResolutionPending` flag so it doesn't queue a second burn before the first split takes effect.
  - **Deviation -- coarse "boundary" heuristic**: only the just-registered cable is treated as the preferred victim; otherwise *some* lowest-tier cable in the network is burned, not necessarily the one geometrically at the junction. In a line like `heavy-heavy-normal-heavy-heavy` this is fine (one burn). In `normal-normal-heavy` it may burn two normals instead of one. Converges either way. Refining this needs the cable grid-adjacency API.
  - **Sharp edge -- retroactive enforcement on load**: with `EnableVoltageTiers` defaulting on, loading *any* existing save that has a normal cable touching a heavy cable (legal in vanilla, possibly common in real bases) will start burning cables on first power tick. The plan (D11) specifies exactly this ("a one-time boundary-cable cascade on first tick"), but it is destructive on first install. Worth reconsidering: on first load, log a warning listing the mixed networks and give the player a grace period, or require a config opt-in for retroactive enforcement. Left as the plan specifies for now.
- **Generators-on-heavy (D13) -- working at the tick level:** `PowerGridTick.CalculateState_New` zeroes a device's generated power if `EnableVoltageTiers` + `Generators Require Heavy Cable` are on, the device is a generator, and the network is not all-heavy. So a solar panel / wind turbine / fuel generator / RTG / Stirling engine / gas-pipe turbine on a normal or super-heavy network produces nothing.
  - **Deviation -- no build-time rejection for generators yet.** A player can still *place* a generator on a normal cable; it just won't generate. The plan wanted a `Device.CanConstruct` postfix rejecting it with a message; that needs the device-to-adjacent-cable lookup at preview time, which wasn't worked out tonight. Add it. (The tick-level enforcement is the real teeth either way.)
  - The "is this a generator?" discriminator (`VoltageTier.IsGenerator`) is a type whitelist: `IPowerGenerator` (covers `SolarPanel`, `WindTurbineGenerator`, `LargeWindTurbineGenerator`) plus `TurbineGenerator`, `StirlingEngine`, `PowerGeneratorSlot` (covers `SolidFuelGenerator`), `PowerGeneratorPipe` (covers `GasFuelGenerator`), `RadioscopicThermalGenerator`. Wireless `PowerReceiver` is **not** in the list (it relays, doesn't generate) -- revisit whether a wireless relay's large output should also be heavy-only.
- **Build-time cable-tier rejection -- best effort only:** `VoltageTierPatches.Cable_CanConstruct_Postfix` tries `CableNetwork.ConnectedNetworks(__instance)` at placement-preview time and rejects with "Cannot connect: different transmission voltage -- use a transformer." if a connected network already carries a different tier. Wrapped in try/catch -- if the preview cable isn't gridded yet (or the API shape differs), it does nothing and the reactive burn-on-join handles it post-placement. Needs in-game verification that `ConnectedNetworks` works on a preview cable.
- **Device-tier policy beyond generators -- NOT implemented.** The plan (D8) also wanted: ordinary devices restricted to normal cable, stationary batteries restricted to heavy, super-heavy as a backbone with no devices at all, and a configurable high-draw-machine whitelist (Carbon Sequester / Advanced Furnace / Arc Furnace) allowed on heavy. None of that placement gating exists yet -- only the cable-tier-incompat rule and the generator rule. Without it, you *can* hang anything off heavy/super-heavy cable; it just isn't lore-correct.
- **Transformer tier mapping (D9/D10) -- NOT implemented.** The plan wanted Small = heavy<->normal only, Medium = heavy<->heavy flow-limiter only, Large = super-heavy<->heavy only, all bidirectional, re-implementing `Transformer.GetGeneratedPower`/`GetUsedPower`/`ReceivePower` symmetrically. Tonight only Re-Volt's transformer *exploit fix* is ported (one-directional, vanilla flow direction). The cable-tier rule already makes a transformer the only thing that can bridge tiers (its input and output sit on separate networks, so no burn fires) -- but any transformer prefab can bridge any tier pair, and transformers are still one-directional. Distinguishing the prefabs needs their prefab names; the bidirectional rewrite is a separate chunk.
- **In-game tier names / rejection message**: the plan (D12) wants "Normal / Heavy / Super-Heavy cable" and the message "Cannot connect: different transmission voltage -- use a transformer". The rejection message string is in the code; no in-game name changes were made.
