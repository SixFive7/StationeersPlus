# Power Grid Plus Playtests

Implemented changes awaiting in-game confirmation. An agent records a playtest here, not in `TODO.md`, when it has changed code whose behavior can only be confirmed by running the game: single-player, a hosted multiplayer session, or the dedicated server under `DedicatedServer/`. This keeps `TODO.md` a list of work still to do, and gives one place to check whether a change already has a pending test before adding another.

Rules:
- Add an entry when code is implemented but its in-game behavior is unconfirmed. Write down everything a tester needs: what changed (commit if there is one), single-player vs multiplayer / dedicated-server, the save or world to set up, the exact in-game steps, what to watch (InspectorPlus request files, specific log lines, on-screen behavior, IC10 reads), and the expected result. Point at any staged `.work/<date>-<slug>/` request files or playbook.
- Check first. Before adding, scan the entries below so a change already covered by a pending test is not duplicated; extend the existing entry instead.
- Remove an entry when one of these happens: a run confirms it works; a run shows it broken (then add a fresh `TODO.md` item for the fix, or keep working on it now); or the player says the playtest is done. Entries are plain bullets, not checkboxes: like `TODO.md`, finished items are removed, not ticked off. Outcomes live in git history.

This mod has never been released (0.2.0 plus the 2026-07-02 power rearchitecture in tree). It compiles and loads cleanly on the dedicated server, and the data / logic half is headless-verified there, including the rearchitecture's allocation math: batteries charge to full across seven wireless links, the conservation checker reports zero violations, the PowerTransmitterPlus billing handshake is live on both sides, and the unknown-bridge census is empty. What remains below is the irreducible client residue: things a headless server physically cannot show (pixel rendering, cursor / settings-panel / dropdown UI, a real second network peer, or an on-screen reading a player would look at).

Environment caution for single-player runs (game 0.2.6403): several third-party Workshop mods are broken by this game version. ForceFieldDoorMod crashes the sim on any world containing a StructureForceFieldDoor (the user's Luna base has 15), and ModularConsoleMod / ScriptedScreens content in older Luna saves fails to deserialize. For Luna-derived tests, use a save stripped of that content (the dedi's `Luna_rearch` was produced this way) or disable the broken mods first.

## Power rearchitecture 2026-07-02: client-visual checks

Commits e9ef3c1f (Stage 1: unified storage-charge flow) and 2a00d674 (Stage 2b: segment adapters + PowerTransmitterPlus interop); Stage 3 (Powered presentation + ledger adoption) is in the working tree. The dedicated-server suites already prove the allocation math; these entries cover only what a player sees. All runnable single-player, or as a client against the dedicated server. A battery-charging base is the setup: solar (or any generator) feeding transformers and/or a linked wireless dish pair, feeding batteries with headroom.

- **Transformer / wireless HUD, tooltip, and IC10 readings now include storage-charge flow.** What changed: routed segments publish rigid plus soft (storage charge) totals, so a bridge that is only charging batteries no longer reads 0. Where to look: a transformer (or linked dish pair) whose downstream is charging batteries with no machines running; hover it and read `PowerActual` (transformer) / the network `RequiredPower` and `PowerPotential` reads from IC10; also the cable-network hover figures on both sides. Expected: the displayed throughput equals the battery charge wattage (plus any machine load), the input side shows that figure times the distance multiplier plus the device's own draw for a wireless pair, and the numbers fall back toward the quiescent draw as the batteries top off. Scripts that assumed "transformer throughput = running machines only" will see higher readings while storage charges; that is the intended new meaning.
- **Healthy idle segmenters read Powered=True (no more phantom "unpowered" chargers).** What changed: Stage 3 Powered presentation (AllowSetPower gate + end-of-tick reconcile on the allocator's healthy set). Where to look: a charger transformer (or dish pair, or APC) that is switched on, fault-free, on a live input network, but idle because its batteries are full. Previously it read Powered=False (dark hover state, IC10 `Power` = 0) and could flicker as demand toggled. Expected: it holds Powered=True while idle-healthy (hover shows powered; IC10 `l rX d0 Power` reads 1) with no flicker across ticks; a segmenter on a genuinely dead input network (night-time solar feed) still goes unpowered with the grey "(No upstream supply)" cue, and a faulted one still goes dark with its fault hover.
- **The negative wireless ledger credit is gone after load (no free-energy startup burst).** What changed: the Stage 3 world-load sweep zeroes every modelled segmenter's saved `_powerProvided`, and a per-tick clamp keeps it in [0, 2x the tick's bill]. Where to look: load a save whose transmitter previously carried a stale negative ledger (the user's Luna save: transmitter 464520 at -176,226; any save from the pre-fix era that showed wireless power flowing "for free" for a while after load). Watch the BepInEx log for the "Ledger adoption sweep: ... segmenter ledger(s) zeroed" lines at first tick. Expected: the pair bills its source normally from the first conducting tick (source-side load rises immediately by delivered times the distance multiplier plus quiescent; no grace period where the source shows no draw), and no recurring "[PowerGridPlus] Ledger clamp" warnings appear afterwards. An InspectorPlus snapshot of both halves' `_powerProvided` should stay within the clamp band during steady transfer.
- **Conservation-check warnings absent in normal play.** What changed: the allocator audits its own grants every tick (`Server - Diagnostics` / `Enable Conservation Check`, default on). Where to look: the BepInEx log / in-game console over a normal session that exercises charging, shedding, an overload, wireless links, and a docked umbilical. Expected: zero "[PowerGridPlus] Conservation violation" and zero "Seg pull invariant violated" warnings. Any occurrence is an allocator bug by definition: capture the full log line (it carries the per-component breakdown) and file it.
