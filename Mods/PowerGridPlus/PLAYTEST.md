# Power Grid Plus Playtests

Implemented changes awaiting in-game confirmation. An agent records a playtest here, not in `TODO.md`, when it has changed code whose behavior can only be confirmed by running the game: single-player, a hosted multiplayer session, or the dedicated server under `DedicatedServer/`. This keeps `TODO.md` a list of work still to do, and gives one place to check whether a change already has a pending test before adding another.

Rules:
- Add an entry when code is implemented but its in-game behavior is unconfirmed. Write down everything a tester needs: what changed (commit if there is one), single-player vs multiplayer / dedicated-server, the save or world to set up, the exact in-game steps, what to watch (InspectorPlus request files, specific log lines, on-screen behavior, IC10 reads), and the expected result. Point at any staged `.work/<date>-<slug>/` request files or playbook.
- Check first. Before adding, scan the entries below so a change already covered by a pending test is not duplicated; extend the existing entry instead.
- Remove an entry when one of these happens: a run confirms it works; a run shows it broken (then add a fresh `TODO.md` item for the fix, or keep working on it now); or the player says the playtest is done. Entries are plain bullets, not checkboxes: like `TODO.md`, finished items are removed, not ticked off. Outcomes live in git history.

This mod has never been released (now at 0.2.0 after the full power rework). It compiles and loads cleanly on the dedicated server (all Harmony patches apply, no errors), and the data/logic half of the rework is now headless-verified. What remains below is the irreducible client residue: things a headless server physically cannot show (pixel rendering, Update-driven animation, cursor / settings-panel / dropdown UI, a real second network peer, or an allocator scenario that needs built topology).

## Single-allocator rework 2026-06-18 (overload-first; Legacy allocator + toggle removed)

The allocator is now a single implementation: the old Legacy allocator, the `EnableSweepAllocator` toggle, and `SweepAllocatorSync` are removed. Overload is evaluated BEFORE shed and is sticky (grow-only within a tick, reset only by the 60 s timeout or a player turn-off); only shed is re-decidable. Verified HEADLESS on the dedicated server this session (do NOT re-test):

- Core no-regression: nets 492209 and 503288 report Required == Actual == Potential every tick, `_isPowerMet=T`, `_powerRatio=1`, no oscillation, no atomic-tick exceptions or non-convergence (`pgp-reversed-transformer-probe`).
- 60-second-freeze fix: in a contended fanout (a low-priority non-step-up consumer X plus an overloading sibling Y, tight feeder), X is NEVER shed and Y is OVERLOADED across the whole trace; a grow-only / shed-first allocator would have frozen X here (`pgp-2cycle-freeze` PASS, F=429369, X=180624, Y=420940).
- OFF-as-reset clears every fault registry (shed / overload / cycle / VVF) for a switched-off device (code-verified, `OffAsResetSweep`).

Live residue (a headless server cannot show the pixels or a real second peer):

## Data-only port tier carve-out + load-regression fix 2026-06-23

Two changes this session, both headless-verified on the dedi against `Luna_pgp_rocket` (current source + the fix), zero Error / Fatal. Do NOT re-test the headless parts.

- **Data-only ports are tier-exempt (the original report).** A device's exclusive data-only port (`NetworkType.Data`, no Power bit) is invisible to the voltage-tier rules and accepts any cable tier; only Power / power+data ports follow the tier rules. Made explicit via `VoltageTier.ConnectionCarriesPower` + `VoltageTier.ReachesNetworkViaPowerPort`, the runtime Misplaced-burn guard in `VoltageTierEnforcer.DetectViolation`, and the cursor's existing `CursorAttachesToPowerPortOf`. ScenarioRunner `pgp-dataport-tier-diag` against the live base: `runtimeInvariant=PASS` (realBugShape=0 across 402 live separate-data-port devices: a device whose data port is on its own network is never in that network's `PowerDeviceList`, so the tier rule and the burn cannot reach it), `cursorCarveOut=PASS` (dataLeak=0 across 72 distinct data ports: `CursorAttachesToPowerPortOf` returns false for every data cable). Covered normal / heavy / superHeavy data networks. Power and power+data enforcement unchanged (8 power-tier would-burn networks still flagged).
- **Load regression fixed.** The 2026-06-22 SDK-style csproj migration auto-globbed in `Patches/AreaPowerControlPassthroughLogicPatches.cs`, a dead file that the old explicit Compile list excluded. Its `[HarmonyPatch(typeof(AreaPowerControl), nameof(AreaPowerControl.CanLogicWrite))]` does not resolve (APC inherits that method from Device rather than declaring it), so `Harmony.PatchAll` threw and PGP loaded with NO patches applied (whole mod dead). Deleted the file (superseded by `InheritedLogicablePassthroughLogicPatches`, which patches the base `Device` logic methods APC inherits). PGP now logs `patches applied`, no fatal.

Remaining live confirm (client-only; a headless dedi cannot render the placement cursor):
- **Data-port cursor ghost.** Single-player, on a device with a separate data port whose power is on tier A (e.g. a Station Battery on heavy, or a normal-powered logic device): pull a DIFFERENT-tier cable (normal, or super heavy) to the DATA port on its OWN run, not laid grid-adjacent to the device's power cable. Expect a GREEN ghost and successful placement; that data network may be any tier. If the data cable is laid touching the device's power cable it merges into the power network and (correctly, per the no-mixing rule) shows the wrong-voltage reject; route it apart. The accept / reject DECISION is headless-proven; this is the on-screen ghost-color confirm only.
