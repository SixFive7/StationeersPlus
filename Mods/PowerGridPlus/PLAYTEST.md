# Power Grid Plus Playtests

Implemented changes awaiting in-game confirmation. An agent records a playtest here, not in `TODO.md`, when it has changed code whose behavior can only be confirmed by running the game: single-player, a hosted multiplayer session, or the dedicated server under `DedicatedServer/`. This keeps `TODO.md` a list of work still to do, and gives one place to check whether a change already has a pending test before adding another.

Rules:
- Add an entry when code is implemented but its in-game behavior is unconfirmed. Write down everything a tester needs: what changed (commit if there is one), single-player vs multiplayer / dedicated-server, the save or world to set up, the exact in-game steps, what to watch (InspectorPlus request files, specific log lines, on-screen behavior, IC10 reads), and the expected result. Point at any staged `.work/<date>-<slug>/` request files or playbook.
- Check first. Before adding, scan the entries below so a change already covered by a pending test is not duplicated; extend the existing entry instead.
- Remove an entry when one of these happens: a run confirms it works; a run shows it broken (then add a fresh `TODO.md` item for the fix, or keep working on it now); or the player says the playtest is done. Entries are plain bullets, not checkboxes: like `TODO.md`, finished items are removed, not ticked off. Outcomes live in git history.

This mod has never been released (now at 0.2.0 after the full power rework). It compiles and loads cleanly on the dedicated server (all Harmony patches apply, no errors), and the data/logic half of the rework is now headless-verified. What remains below is the irreducible client residue: things a headless server physically cannot show (pixel rendering, Update-driven animation, cursor / settings-panel / dropdown UI, a real second network peer, or an allocator scenario that needs built topology).

- **Fault flash colors + countdown.** Place an IC10 housing + chip on transformer 360223's data port. `s d0 Setting 10` (below its ~50 W downstream): RED overload flash on the on/off button + hover "(Overloaded: Downstream demand exceeds this transformer's limit! Hover content and colors are already verified; this is the visible-pulse + smoothness check.
- **Flash visible on each segmenter class; NOT on Battery/APC.** Force a fault on one of each (IC10 Setting throttle on a transformer; for the others the campaign confirmed the registry path) and watch the on/off button. ### Group B: same save, a few built structures (single-player)
- **Cycle-fault flash.** Build an anti-parallel transformer pair (T1 input N1 / output N2, T2 input N2 / output N1) on a powered grid; also try a transformer+battery+APC ring. Every member flashes RED, no cable burns, downstream goes dark; after 60 s it retries. Detection produced zero false positives on the baseline, so this confirms the flash + no-burn behavior.
- **Battery-overload + two-battery synced recovery (P5).** A battery-fed room with no other supply: drain below the rigid load, expect a RED overload flash and a dark subnet (not partial power), re-firing each 60 s. Then two batteries on one output net, each discharge-capped too low alone but jointly enough: they share and do NOT flash; raise the load above their combined cap so both flash, drop it back, they must recover TOGETHER in one re-arm; with the load still high, toggling one OFF/ON must NOT recover and must not strand the net on a drifting timer. InspectorPlus: `OverloadRegistry` entries on the cohort must share one expiry tick.
- **Dead-input recovery timing (P7).** A transformer (also an APC, and a PT/PR pair) feeding a load with its input cable on a network with NO source: grey "(No upstream supply)" cue, no flash, no countdown; connect a generator to the input and it must start delivering THE SAME tick (no 60 s lockout). Cue content already verified; this is the same-tick recovery + no-pulse check.
- **Producer-rich coverage (P10 / P11 / P12 instance gaps).** The campaign save had only solar + a solid generator + a connector, so VVF logic-read was instance-confirmed on solar and the solid generator only. Build one of each remaining producer (gas generator, stirling engine, small wall turbine, small + large wind turbine) wired straight to a consumer with no transformer, and confirm: each enters VVF; a Logic Reader / IC10 `l rX d0 VariableVoltageFault` returns 1 (solar / wind / turbine / gas / stirling), while the connector and RTG stay hover-only; the gas + stirling generators show the RED button flash (the solid one was confirmed to render, 2 renderers). Then the multi-producer network-retry: put a buttonless producer (solar) and a buttoned one (gas/solid) on the SAME no-transformer net, confirm their VVF countdowns tick down TOGETHER (one synced timer), add a transformer and toggle the buttoned producer OFF, and confirm BOTH clear that tick (the buttonless one has no button of its own). The single-device OFF-as-reset and active-source gate are already verified headless; this confirms the multi-producer cohort firing + the per-instance flash render.
- **Live wrong-tier burn + cursor rejects (P1 / NEW-3).** Build a normal cable bridging into a powered heavy network: the boundary cable bursts to wreckage within a frame or two, the network splits, no cascade on a multi-junction build. Hover the wreckage: "Burned: Wrong voltage ...". Hold a normal-cable coil next to a heavy cable / an RTG: red ghost + the wrong-voltage reject message. ### Group C: PowerTransmitter links (single-player; allocator decisions are server-side)
- **P6 input-limited link SHEDS, link-rated link OVERLOADS.** Feed a transmitter's input from a small source whose PotentialLoad is below both the link cap and downstream demand: the dish shows SHED (orange, "(Shedding: Insufficient upstream supply!)"), receiver subnet dark; adding input supply clears it.
