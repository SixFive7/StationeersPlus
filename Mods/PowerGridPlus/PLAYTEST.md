# Power Grid Plus Playtests

Implemented changes awaiting in-game confirmation. An agent records a playtest here, not in `TODO.md`, when it has changed code whose behavior can only be confirmed by running the game: single-player, a hosted multiplayer session, or the dedicated server under `DedicatedServer/`. This keeps `TODO.md` a list of work still to do, and gives one place to check whether a change already has a pending test before adding another.

Rules:
- Add an entry when code is implemented but its in-game behavior is unconfirmed. Write down everything a tester needs: what changed (commit if there is one), single-player vs multiplayer / dedicated-server, the save or world to set up, the exact in-game steps, what to watch (InspectorPlus request files, specific log lines, on-screen behavior, IC10 reads), and the expected result. Point at any staged `.work/<date>-<slug>/` request files or playbook.
- Check first. Before adding, scan the entries below so a change already covered by a pending test is not duplicated; extend the existing entry instead.
- Remove an entry when one of these happens: a run confirms it works; a run shows it broken (then add a fresh `TODO.md` item for the fix, or keep working on it now); or the player says the playtest is done. Entries are plain bullets, not checkboxes: like `TODO.md`, finished items are removed, not ticked off. Outcomes live in git history.

This mod has never been released (0.2.0 plus the 2026-07-02 power rearchitecture in tree). It compiles and loads cleanly on the dedicated server, and the data / logic half is headless-verified there, including the rearchitecture's allocation math: batteries charge to full across seven wireless links, the conservation checker reports zero violations, the PowerTransmitterPlus billing handshake is live on both sides, and the unknown-bridge census is empty. What remains below is the irreducible client residue: things a headless server physically cannot show (pixel rendering, cursor / settings-panel / dropdown UI, a real second network peer, or an on-screen reading a player would look at).

Environment caution for single-player runs (game 0.2.6403): several third-party Workshop mods are broken by this game version. ForceFieldDoorMod crashes the sim on any world containing a StructureForceFieldDoor (the user's Luna base has 15), and ModularConsoleMod / ScriptedScreens content in older Luna saves fails to deserialize. For Luna-derived tests, use a save stripped of that content (the dedi's `Luna_rearch` was produced this way) or disable the broken mods first.

## Decision-33 phase 2, 2026-07-17: the overload split by cause (build 34DBD844, in tree)

What changed: the structural Device overloaded rule split per the decision-33 hybrid. Rule 1a (per round, immediate, grow-only) now trips only when a network's OWN rigid demand, the machines wired directly to it, exceeds the hardware cap: shedding can never relieve that. The forwarded-demand case moved to rule 1b (DetectResidualOverload), evaluated at allocation-loop settle where every shed is final: the UNSHEDDABLE residual (own machines plus still-active child claims) against the same cap, with an outer re-decide iteration so an overload that frees upstream budget still un-sheds its victim the same tick (the 2c guarantee, preserved). The capacity sum in both rules now counts Deprioritized-but-not-Locked suppliers at their hardware cap, so "Device overloaded" always means demand exceeds what the hardware could deliver even fully fed, and a same-tick shed of one supplier can no longer stamp a false sticky overload on its surviving co-supplier. Fixture lockstep: pgp-overload-split-fixture gained P2c (1a silent on forwarded demand, 1b owns the trip with the residual payload) and P2d (shed supplier's cap stays in the sum, no false stamp), and its P4b2 hover assert was updated from the retired "Drawing X of Y" wording to the 2026-07-15 overhaul strings (it was failing stale against the correct render). Dedi-verified on fresh Lunar worlds: pgp-overload-split-fixture 14/14, pgp-chain-fixture regression 14/14 (leaf-first order, hop protection, lockout countdown/expiry/re-engage all intact; overloadHits=0 on the availability-short chain), zero errors both runs.

- **Machine-driven overload still instant.** Load a transformer's output net with machines alone beyond its Setting: the red Device overloaded fault appears immediately with the "Downstream demand of X exceeds the available Y" hover, exactly as before.
- **Shedding gets its chance before a trunk faults.** Starve a feeder whose output net carries sub-transformers with sheddable branches: the lowest-priority branch sheds (orange, contest numbers) and the feeder stays healthy when shedding fits the residual under its cap; the feeder trips red only when even full shedding cannot fit. On the Luna base, the 511091/511092 pair shape should now shed leaves instead of red-faulting the pair, wherever sheddable leaves exist.
- **No false red on a shed sibling's co-supplier.** When one of two parallel suppliers sheds (its own input starved), the survivor must NOT show "Device overloaded" out of nowhere; the demand it cannot cover falls through to the availability story instead.

## Decision-33 phase 3, 2026-07-17: watt-minimizing victim selection (build 950D3BA6, in tree)

What changed: the deprioritization victim selector's within-tier rule. Tiers still go priority ASC; a tier whose combined claims cannot cover the remaining deficit is shed wholesale (LowerPriority reasons, declining per-victim shortfall), and the first tier that CAN cover now ends selection with the victim set of the SMALLEST TOTAL WATTS whose sum still covers the deficit (exact subset choice within DP bounds, the legacy best-fit walk beyond them; on equal claims the lowest ReferenceId is shed, the pinned legacy tie-break). This kills the mass-outage bias: an 8 kW deficit against a 50 kW branch plus two small rooms sheds the rooms (11 kW total), never the branch. The DeprioritizeReason vocabulary collapsed to two values (LowerPriority / EqualWattCover) with the hover Reason block rewritten to match; the wire byte and join-suffix shape are unchanged. Dedi-verified on fresh Lunar worlds: pgp-deprioritization-victim-fixture 17/17 (all sixteen legacy cases reproduce exactly; the new P14 pins the divergence, shedding 6000+5000 over the 50000 cover), pgp-chain-fixture 14/14 and pgp-overload-split-fixture 14/14 regressions, zero errors across all three runs. POWER.md sections 8.0.3 / 8.3 / 8.3.3 / 11.1 body text synced to the new policy.

- **Watt-min shed in game.** Build a net feeding one large branch (through a sub-transformer) plus two small rooms at equal priority; starve it by a small deficit: the two small rooms shed (orange, both showing the same shortfall and the smallest-combined-cut reason) and the large branch stays lit. Under the old rule the large branch died alone.

## Decision-33 phase 4, 2026-07-17: deep-first shedding and sheddable step-ups (build 9085456C, in tree)

What changed: three engine changes completing the shedding redesign. (1) The forward pass no longer decides victims: each round grants with the previous decisions, DecideVictimsDeepFirst re-decides leaf-to-source against the settled grants, and a second forward pass settles with the fresh decisions, so Unmet and the flows the supply-overload rules read are never a pre-shed transient. (2) The backward desire gate now excludes Deprioritized contributors, so a shed branch stops inflating its parents' claims the very next round and leaf-to-trunk escalation completes within one tick (previously one level per TICK, each level darkening the source net for a spurious 60 s hold); re-decidability rides the decision pass's flag clear, and the stranded-inflow clawback stays as the belt for 2-cycle-union settles. (3) The categorical step-up victim exclusion is retired: protection is FeedsActiveSeg alone, so a terminal step-up branch (a heavy machine net fed by small step-ups) joins leaf-first shedding as the leaf it functionally is, while a trunk step-up stays protected exactly while its subtree is active; the Stationpedia deprioritization footer now tells players to give backbone transformers a high Priority. Dedi-verified on fresh Lunar worlds: pgp-chain-fixture 14/14 (exact lowest-priority-first leaf order, hops never victims, no-load trunk clean, lockout countdown/expiry/re-engage intact), pgp-deprioritization-victim-fixture 17/17, pgp-overload-split-fixture 14/14, and pgp-fresh-device-trace clean (stub DEAD_NOSUPPLY, merged net LIVE on the first post-bridge tick, neighbors never flicker); zero errors across all four runs.

- **Terminal step-up sheds as a leaf.** A heavy machine net fed by a small step-up from a shared normal net; starve the normal net: the step-up branch now competes on Priority and watt-minimization like any room (orange with the contest numbers when it loses) instead of being invisibly protected while innocent rooms shed around it and both nets went dark uncued.
- **Priority dial works on backbones.** Raise the Priority of a trunk transformer that feeds further transformers; a deficit must shed low-priority rooms and never the raised trunk.
- **No source blackout during escalation.** Starve a two-level chain hard enough that a leaf AND its feeder must both shed: the source net's own machines stay lit throughout (previously the source went dark whole for 60 s at each escalation level).

Known follow-up (not blocking the shipped mod, which builds 0/0): the ScenarioRunner dev harness (`DedicatedServer/dev-plugins/ScenarioRunner`, not shipped) has fixtures that reflection-pin the OLD registry signatures (OverloadSplitFixture's 4-arg `NoteOverload`, the Deprioritization P3c 5-arg `NoteDeprioritized` + 4-out `TryGetFault`, the MP P1 tuple shapes); they report FAIL until updated to the widened signatures. ## Decision-33 phase 5, 2026-07-17: Undersupplied face, battery rate-vs-energy split, gauge honesty (in tree)

What changed: three player-facing pieces closing the decision-33 fault-story gaps. (1) The rule-2 elastic capacity fault now distinguishes WHY a store fell short: a store limited by its discharge-rate hardware (or the APC's rate-vs-cable side) still takes the red Device overloaded lockout, but a store that simply ran out of stored energy is skipped (new `Elastic.StoredBound`), so a drained battery or APC cell no longer red-faults as if the hardware were undersized; its shortfall falls through to the availability story instead. (2) That availability story now has a face: every net whose verdict lands DEAD_UNMET is marked in the new `UndersuppliedRegistry` (needs W, delivered W, strongest unlocked feeder), synced to clients as snapshot kind 6 with the DeadInput-style keep-alive, and every device on such a net hovers the amber "Undersupplied" info block: "Needs X while upstream delivers Y" plus "Check {feeder}". It is an info state, not a lockout: no flash, no countdown, no fourth fault kind. (3) Gauge honesty: `WriteBack` zeroes a network's `CurrentLoad` readout when the verdict is not LIVE, so cable analysers and network hovers on a dark net read 0 instead of the phantom model flow (the internal flow is kept for fuse and burn logic). Also in the same build: the tick-duration watchdog gained a two-tick cold-start grace after each load boundary, because the first tick JIT-compiles the whole allocator pipeline (measured ~950 ms allocator span, once, on a FRESH EMPTY world) and was tripping the "performance regression, please report it" warning on every load; a plain-soak re-run on the grace build ticked clean with zero watchdog lines. Dedi-verified on fresh Lunar worlds: pgp-overload-split-fixture 15/15 (new P3d pins the rate/energy split), pgp-chain-fixture 14/14, pgp-deprioritization-victim-fixture 17/17, pgp-fresh-device-trace tick-exact against the phase-1 baseline (dark only in the designed build windows, merged net LIVE first post-bridge tick, neighbors never flicker).

- **APC discharge 0-default (2026-07-18): the cell matches the wire.** With `APC Battery Discharge Rate` at its new default 0, the cell discharges up to the output wire's tier cap (5000 W on normal cable) and never above it. On the workshop APC (ref 19765): the 1115 W room now runs from the cell instead of red-faulting at 1.0 kW, and when the cell drains below demand the room ends in the amber Undersupplied face (the designed drained-store ending). The value 0 was written into both the client and dedicated-server config files; any other machine's saved config keeps its old 1000 until hand-edited or reset.
- **Drained APC shows Undersupplied, not Device overloaded.** An APC with no upstream feed and its cell run down below the room's rigid demand: previously a red Device overloaded lockout (blaming the hardware); now the amber Undersupplied block with the needs-vs-delivers numbers. Force the OTHER case (cell charged, demand above the APC's 500 W discharge rate or its output cable cap): the red Device overloaded fault still fires.
- **Undersupplied hover on a starved room.** Starve a room past what shedding can cover (kill generation, no sheddable branches left): every device on the dark net hovers the same amber "Undersupplied" block, needs red, delivers blue, plus "Check {name}" naming the strongest feeder into the net. The 60 s hold rhythm is unchanged; the block is the explanation the dark room never had.
- **MP sync.** With a client connected, the client hovers the same Undersupplied numbers as the host on the same net; a client joining mid-state sees it within a couple of ticks (keep-alive kind, not in the join suffix).
- **Dead net reads 0 flow.** Cable analyser / network hover on a held-dark net: `CurrentLoad` reads 0 while the net is dark, jumps to the real flow the tick it relights (previously showed the would-be flow while dark).
