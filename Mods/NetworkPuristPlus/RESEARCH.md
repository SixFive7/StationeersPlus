# Network Purist Plus Research

Durable, project-scoped internals: architecture, patch catalog, the game internals this mod relies on, and the design decisions behind it. The deep game-internals writeup of the long-variant system lives in the central knowledge base at `Research/GameClasses/MultiMergeConstructor.md`; this file only restates what is load-bearing for this mod.

## What "long pieces" are in Stationeers

Stationeers ships "long" straight building pieces: extra prefabs of the ordinary `Pipe` / `Piping` / `Cable` / `Chute` classes whose mesh/collider footprint spans 3, 5, or 10 small-grid cells. There is no per-length C# class; the length is baked into the prefab asset. They live in `MultiMergeConstructor` kits (`ItemKitPipe`, `ItemKitChute`, the super-heavy cable kit) and you reach them either by picking the variant from the kit's mouse-wheel option list, or, with the kit's `ToolExit` item in your off hand, by merge-absorbing an existing straight run.

Complete set (verified against the game's `english.xml` `<Key>...Straight[0-9]+</Key>` keys and the in-game changelog):

- `StructurePipeStraight3` / `5` / `10`
- `StructureInsulatedPipeStraight3` / `5` / `10`
- `StructurePipeLiquidStraight3` / `5` / `10`
- `StructureInsulatedPipeLiquidStraight3` / `5` / `10`
- `StructureChuteStraight3` / `5` / `10`
- `StructureCableSuperHeavyStraight3` / `5` / `10` (and the `...Burnt` electrocution-damage-state siblings of each, which are `CableRuptured` prefabs)

No long walls, girders, beams, rails, ducts, or low-volume pipes exist. `StructureConveyorStraightLong` belongs to the old Recurso "Flexi-belt" conveyor-belt prefab family (`StructureConveyorStraight` / `StraightLong` / `StraightShort` / `Corner` / `Riser`, plus the `ItemKitConveyor` kit and `DynamicCrateConveyorSupplies` crate), which is **cut content**: the prefabs and string / Stationpedia entries still ship registered, but no conveyor-specific C# class exists (`Assembly-CSharp` has only leftovers: `LoreFactions.Recurso`, a "Conveyors" audio bucket, `CrateType.ConveyorSupplies`, the never-set `DynamicThing.OnConveyor` bool, the chute-related `DeviceImportExport.ImportConveyorPosition` transforms). The kit builds nothing, and the pieces are reachable only via the creative-mode prefab spawner; "building" one errors. So it is not a `MultiConstructor` kit option and is not merge-able. Out of scope for this mod, and if it is ever brought in, the work is "hide a vestige" (set `HideInStationpedia`, drop from the creative spawn list), not the `Constructables`-strip + merge-reject + on-load replace this mod does for the functional long pipe / chute / cable variants. Full writeup: `Research/GameSystems/ConveyorBeltCutContent.md`.

## Architecture

`LaunchPadBooster` is a dependency (referenced for the `Mod` / `Networking` / `IJoinValidator` / `INetworkMessage` API surface; it ships with StationeersLaunchPad, `<Private>False</Private>`). Three themes share one plugin and one Harmony instance:

1. **Long-variant removal** (v1.0): hide the long prefabs, strip them from the build kits, rebuild placed long runs on load, rewrite mid-game long-piece placements into single tiles.
2. **Cable alignment** (v1.1): re-roll every straight cable to one canonical orientation per run axis, both on load (existing worlds) and as cables are built (new placements), and switch the straight-cable build cursor from the 24-orientation `Exhaustive` cycle to the 3-orientation `Straight` cycle. The cursor-`AutomaticSetup` half also runs on the single-tile straight pipe and chute base prefabs (see "why all four building blocks" and the `MultiMergeConstructor.Construct` pitfall) -- the *roll*-normalisation half stays cable-only.
3. **Settings + network enforcement** (v1.1): a `Server - *` settings panel (master toggle + per-family long-piece toggles + cable-alignment toggle, all server-authoritative -- see `Settings.cs`), server-synced via a `SettingsConfigMessage` + `IJoinSuffixSerializer`, and a `LaunchPadBooster` `MOD.Networking` setup (`Required = true` + an `IJoinValidator`) that rejects a joining client whose mod-version OR settings differ from the host's. The settings decide which `Constructables` get stripped / which prefabs get `HideInStationpedia` / which bases get the build-cursor `AutomaticSetup`, all at `Prefab.OnPrefabsLoaded` -- before any join -- so each machine first applies its own config; the `JoinValidator` is what keeps everyone's build-kit option lists and Stationpedias in sync. (See `Research/Patterns/ConflictDetection.md` for the broader StationeersLaunchPad-mod join-handshake pattern and `Research/Protocols/LaunchPadBoosterNetworking.md` for the `IJoinValidator` / `INetworkMessage` mechanics.)

Parts:

- `Plugin.cs` (`NetworkPuristPlusPlugin`, implements `IJoinValidator` + `IJoinSuffixSerializer`): BepInEx entry point. `Awake()` binds the settings (`Settings.Bind`) and wires the host-side settings broadcast (`SettingsConfigSync.HookHostBroadcast`), then subscribes to `Prefab.OnPrefabsLoaded`. On that callback it sets up `LaunchPadBooster` networking (`MOD.Networking.Required = true`, `RegisterMessage<SettingsConfigMessage>()`, `JoinValidator = this`, `JoinSuffixSerializer = this`), builds the registry (which also runs the build-cursor `AutomaticSetup` pass), and `Harmony.PatchAll()`s (picks up all the `[HarmonyPatch]` classes below). The `IJoinValidator` (de)serialises the eight settings bools and rejects a mismatch on the first differing setting with a readable per-setting error; the `IJoinSuffixSerializer` pushes the host's eight bools into the joiner's world-snapshot for the joiner's client-side mirror (the validator already guaranteed the values match). It also defines the `PlayerLog` / `PlayerWarn` / `PlayerError` helpers, which log to BepInEx + `Debug.Log` + the in-game `~` console (`Assets.Scripts.ConsoleWindow`, brought in via a `using ConsoleWindow = Assets.Scripts.ConsoleWindow;` alias -- an alias rather than `using Assets.Scripts;` because that namespace also has a `Settings` type that would clash with `NetworkPuristPlus.Settings`; each `ConsoleWindow.Print*` call is wrapped in a try/catch because some fire from `OnPrefabsLoaded` before the console UI is up).
- `Settings.cs`: static. The eight `ConfigEntry<bool>` entries (`Enable Network Purist Plus` master; `Remove Long Gas Pipes` / `... Liquid Pipes` / `... Insulated Gas Pipes` -- insulated gas pipe only -- / `... Insulated Liquid Pipes` / `... Chutes` / `... Super-Heavy Cables` per-family; `Align Straight Cables`), all in `Server - Pieces` / `Server - Cables` sections with `Order` tags, `(Server-authoritative)` description prefixes, and a `("RequireRestart", true)` tag (each gates `Prefab.OnPrefabsLoaded`-time work, so a change needs a game restart -- StationeersLaunchPad surfaces a restart indicator from the tag; mirrors PowerTransmitterPlus / PowerGridPlus). The C# field names mirror the keys (`RemoveLongGasPipes`, `RemoveLongInsulatedGasPipes`, ...). The pipe-family setting names spell out "Gas Pipe" / "Insulated Gas Pipe" for the player even though the game's own term for the basic pipe is just "Pipe" (and "Insulated Pipe" is the insulated *gas* pipe) -- otherwise the panel doesn't distinguish the gas toggle from the liquid one. `LongPieceFamily` enum (six families: `GasPipe` / `LiquidPipe` / `InsulatedPipe` / `InsulatedLiquidPipe` / `Chute` / `SuperHeavyCable`, plus `Unknown`) + `ClassifyFamily(baseName)` (substring match on the de-numbered base name, more-specific strings tested first -- `"InsulatedPipeLiquid"` before `"InsulatedPipe"` before `"PipeLiquid"`) + `FamilyEnabled(family)` + `MasterEnabled` / `CableAlignmentEnabled` convenience accessors. (The enum + `ClassifyFamily` are keyed off prefab-name substrings, not the setting names, so the rename didn't touch them.)
- `SettingsConfigMessage.cs`: an `INetworkMessage` carrying the eight bools; `FromLocalConfig()` builds it host-side; `Process(hostId)` (client side only) hands it to `SettingsConfigSync.OnHostConfigReceived`. Wire format: the first seven slots are stable (`Enabled` / `RemoveLongGasPipes` / `RemoveLongLiquidPipes` / `RemoveLongInsulatedGasPipes` / `RemoveLongChutes` / `RemoveLongSuperHeavyCables` / `AlignStraightCables`); `RemoveLongInsulatedLiquidPipes` is the 8th bool, appended last so the read order stays stable. (The v1.1-polish rename of the gas-pipe toggles -- `RemoveLongPipes` -> `RemoveLongGasPipes`, `RemoveLongInsulatedPipes` -> `RemoveLongInsulatedGasPipes` -- only changed names, not slot positions, so the wire format is unchanged. The `IJoinValidator` payload in `Plugin.cs` uses the same eight-bool order.)
- `SettingsConfigSync.cs`: static. Holds the host's last-known settings as seen on a client (logging / diagnostics only); `HookHostBroadcast()` wires a `SettingChanged` handler on each entry to `BroadcastIfHost()` (which `SendAll`s a fresh `SettingsConfigMessage` when `NetworkManager.IsServer`); `OnHostConfigReceived(msg)` updates the mirror and logs on change.
- `LongVariantRegistry.cs`: static. `Build()` short-circuits when `Settings.MasterEnabled` is false; otherwise scans `Prefab.AllPrefabs`, classifies the long variants, **skips a long variant whose family's per-family toggle is off** (leaving it as vanilla), maps each accepted long variant to its single-tile base prefab and segment count, records the long-variant prefab object in `LongVariants` (for the Stationpedia re-hide patch), sets `HideInStationpedia` on each, strips the accepted long variants out of every `MultiConstructor.Constructables` list, and runs `SmartRotate.AutomaticSetup` on the single-tile straight pipe/cable/chute base prefabs whose long variants it strips (the merge-with-tool fix; per-family gated via `LongToBase.Values`) plus -- only if `Settings.CableAlignmentEnabled` -- every straight cable of every tier (the cursor half of the alignment feature for cable tiers; the stripped super-heavy-cable base is already in the first set).
- `ReplaceLongPiecesOnLoadPatch.cs`: a Harmony postfix on `World.OnLoadingFinished` that, on the host / single-player, rebuilds every placed long run from single-tile pieces (cable singles spawned at `CableRoll.Canonical`). Early-returns when `!Settings.MasterEnabled` (and naturally a no-op when `LongToBase` is empty).
- `RewriteLongVariantOnConstructPatch.cs`: a Harmony prefix on `Constructor.SpawnConstruct(CreateStructureInstance)` that rewrites a long-variant placement into N single-tile placements as it is built. Early-returns `true` when `!Settings.MasterEnabled` (and naturally falls through when `LongToBase` is empty).
- `CableRoll.cs`: static helper. The canonical-roll set, run-axis classification, the `IsNormalisableStraight` predicate, and the verified-minimal runtime re-roll recipe.
- `NormalizeCableRollOnLoadPatch.cs`: a second Harmony postfix on `World.OnLoadingFinished` that sweeps `GridController.AllStructuresPool` and re-rolls every placed straight cable. Early-returns when `!Settings.CableAlignmentEnabled`.
- `NormalizeCableRollOnRegisterPatch.cs`: a Harmony postfix on `Cable.OnRegistered(Cell)` that re-rolls a freshly built straight cable the instant it registers. Early-returns when `!Settings.CableAlignmentEnabled`.
- `HideLongVariantsStationpediaPatch.cs`: a Harmony postfix on `SPDADataHandler.HandleThingPageOverrides()` (the global-namespace `SPDADataHandler` type; `Stationpedia.DataHandler` is a static field of it) that re-asserts `HideInStationpedia = true` (and the parallel `SPDADataHandler.HiddenInPedia` dictionary entry) on every long-variant prefab in `LongVariantRegistry.LongVariants`, after the XML page-override pass has run, in case a future mod's page override would clear the flag. Gated on `Settings.MasterEnabled`; a no-op when `LongVariants` is empty. See `Research/GameClasses/Stationpedia.md` for `HandleThingPageOverrides` / `PopulateLists` ordering.

## File walkthrough

- `Plugin.cs` -- `Awake()` only stores the logger and registers the `Prefab.OnPrefabsLoaded` handler (StationeersLaunchPad loads mods progressively, so deferring to that event makes the scan see mod-added prefabs and runs after each prefab's `OnPrefabLoad` and before the Stationpedia is built). `OnPrefabsLoaded()` calls `LongVariantRegistry.Build()` then `new Harmony(PluginGuid).PatchAll()`. The success/failure log line covers both themes (the cable-alignment patches run regardless of whether any long-variant prefab was found).
- `LongVariantRegistry.cs` -- `LongToBase` (`Dictionary<int, Structure>`: long-variant `PrefabHash` -> single-tile base prefab), `LongToCellCount` (`Dictionary<int, int>`: long-variant `PrefabHash` -> segment count from the trailing digit run; diagnostic only -- block 4 reads the actual footprint off `GridBounds`). `Build()`: pass 1 walks `Prefab.AllPrefabs`, matches each `Structure` name against `^(Structure[A-Za-z]+Straight)(\d+)(Burnt)?$`, derives the base name as group 1 + group 3, requires `Prefab.Find(baseName)` to be a registered `Structure`, records the mapping + segment count, sets `HideInStationpedia = true`. (No footprint gate any more -- the comment block was corrected: `Structure.GridBounds` *is* populated by `CachePrefabBounds()` during `Prefab.Register`, so `CellCount(structure)` returns real counts at `OnPrefabsLoaded`, but the name pattern alone suffices and the on-load rebuild reads the footprint off the placed instance regardless.) Between passes it builds `autoSetup`, a `HashSet<int>` of prefab hashes to give the `Straight` connection type: every `LongToBase.Values` prefab (the single-tile bases whose long variants are stripped -- the pipe families, the chute, the super-heavy cable) plus every `Cable` in `Prefab.AllPrefabs` with `IsStraight` (all cable tiers, including the normal/heavy ones that have no long variant). Pass 2 walks `Prefab.AllPrefabs` again for `MultiConstructor` (covers `MultiMergeConstructor` by inheritance), `RemoveAll`s the long variants from `Constructables`, clamps `LastSelectedIndex`, and calls `AutomaticSetupStraight` on each remaining `Constructables` entry whose hash is in `autoSetup`. Pass 3 walks `Prefab.AllPrefabs` once more and calls `AutomaticSetupStraight` on every prefab whose hash is in `autoSetup`. `AutomaticSetupStraight(Thing)` is `if (!(thing is ISmartRotatable rotatable)) return false;` then a try/catch wrapper around `SmartRotate.AutomaticSetup(rotatable)`; returns true on success. `CellCount(Structure)` is the footprint helper; returns 0 on any failure.
- `ReplaceLongPiecesOnLoadPatch.cs` -- postfix on `World.OnLoadingFinished`. Bails if `!GameManager.RunSimulation` (client) or the registry is empty. Snapshots `GridController.AllStructuresPool` into a list (the pool is mutated by the destroy/create loop). For each long piece: computes its small-grid cells via `GridBounds.GetLocalSmallGrid(ThingTransformPosition, ThingTransformRotation)`, captures rotation / owner / paint swatch index, picks `spawnRotation = (basePrefab is Cable bc && bc.IsStraight) ? CableRoll.Canonical(rotation) : rotation` (a rebuilt straight-cable run is born canonical; pipe/chute singles keep the long piece's roll), `OnServer.Destroy`s the long piece, then `Constructor.SpawnConstruct(new CreateStructureInstance(basePrefab, cell, spawnRotation, owner, colorIndex))` for each cell. `PaintedColorIndex` returns the swatch index only when `CustomColor.Normal != PaintableMaterial` (a real paint override), else `-1`.
- `RewriteLongVariantOnConstructPatch.cs` -- prefix on `Constructor.SpawnConstruct(CreateStructureInstance)`, signature `Prefix(CreateStructureInstance instance, ref Structure __result)`. If `instance.Prefab` is not a long variant (`!LongToBase.TryGetValue(instance.Prefab.PrefabHash, ...)`) it returns true (run the original). Otherwise it reads the footprint cells off `instance.Prefab.GridBounds.GetLocalSmallGrid(instance.WorldPosition, instance.WorldRotation)`; if that comes back null or `< 2` it warns and returns true (lets the long piece be placed -- the on-load rebuild is the backstop). Otherwise it picks `rot = (basePrefab is Cable bc && bc.IsStraight) ? CableRoll.Canonical(instance.LocalRotation) : instance.LocalRotation`, loops `Constructor.SpawnConstruct(new CreateStructureInstance(basePrefab, cell, rot, instance.OwnerClientId, instance.CustomColor))` once per cell, sets `__result` to the first single (most callers discard it; the one `MultiMergeConstructor` pipe-merge caller that reads it gets a valid `Pipe`), and returns false. The inner `SpawnConstruct` calls fall through this prefix because `basePrefab` is the single-tile base, not a long variant -- no recursion. Wrapped in try/catch; on any exception it returns true so the original placement proceeds.
- `CableRoll.cs` -- `CanonZ` = `Quaternion.identity` (Z run: forward +Z, band/local-up on world +Y -- the orientation a vanilla fresh-cursor Z straight comes out at), `CanonX` = `Quaternion.Euler(0,90,0)` (X run: forward +X, band on world +Y), `CanonY` = `Quaternion.Euler(270,0,90)` (Y run: forward +Y, band on world -X). One canonical rotation per run axis: Z is the fresh-cursor orientation; X and Y are the per-axis plurality corner-mating optimum (the canonical roll whose band face the largest single share of corner cables on that axis exposes their band to -- see `Research/GameSystems/PlacementOrientation.md`, "Why straight-cable roll normalisation does not fix the band-seam at a corner cable"). **This is not the old framing** -- the v1.1-polish change moved the canonical set off `RotationsList[SmartRotate.ConnectionType.Straight]` (which was `Euler(0,0,90)` / `Euler(0,90,0)` / `Euler(90,0,0)`, the set that mated with the *fewest* corners) onto the per-axis plurality optimum, which also fixes the reported 318512/318557 case. Changing these constants triggers one more one-time re-roll of every straight cable on the next world load -- expected, cosmetic-only, persists in the save, syncs to clients. `Canonical(Quaternion)`: classifies the run axis from the dominant absolute component of `rotation * Vector3.forward` and returns the matching canonical rotation. `IsCanonical(Quaternion)`: `Quaternion.Angle(rotation, Canonical(rotation)) < 1f`. `IsNormalisableStraight(Cable c)`: `c != null && !c.IsBeingDestroyed && c.IsStraight && !LongVariantRegistry.LongToBase.ContainsKey(c.PrefabHash)`. `Normalise(Cable c)`: if `IsNormalisableStraight` and the current rotation is more than 1 deg off canonical, sets `c.ThingTransform.rotation = canon` (visible mesh follows that frame), `c.RegisteredRotation = canon` (what the save records), `c.Direction = canon` (the network-sync-only join-package field), and on the server `c.NetworkUpdateFlags |= 1` (bit 1 = Thing transform delta to connected clients); returns true if it changed. It does NOT re-register the cable or call `RebuildGridState` -- the occupied cell of a single-tile `SmallSingleGrid` is rotation-invariant, and a re-register would re-run `Cable.OnRegistered` -> `CableNetwork.Merge`. **Known limitation:** this does not fully eliminate the band-seam at corner cables -- corner pieces have a fixed band orientation the mod doesn't touch (re-rolling a corner is connectivity-relevant, its open ends are off the run axis), no single per-axis roll mates with every corner, and a straight between two corners with contradictory band faces can match at one end only; this choice mates with the most. A full corner-seam fix (re-roll each corner-adjacent straight per its specific corner) is a separate, larger feature -- see `TODO.md`.
- `NormalizeCableRollOnLoadPatch.cs` -- second postfix on `World.OnLoadingFinished` (alongside `ReplaceLongPiecesOnLoadPatch`; order irrelevant -- see the patch catalog). Bails if `!GameManager.RunSimulation`. Snapshots `GridController.AllStructuresPool` for every `Cable` matching `IsNormalisableStraight` (skips long cables and `IsBeingDestroyed` -- the long rebuild's deferred-destroy targets), then `Normalise`s each in a per-cable try/catch and logs the count via `PlayerLog`.
- `NormalizeCableRollOnRegisterPatch.cs` -- postfix on `Cable.OnRegistered(Cell)`, signature `Postfix(Cable __instance)`. Bails if `!GameManager.RunSimulation || GameManager.GameState == GameState.Loading` (host/single-player only, not save-deserialise). Otherwise `CableRoll.Normalise(__instance)` in a try/catch. No per-cable log here (it would spam on a normal load where every cable registers); the on-load sweep does the summary logging.

## Patch catalog

- `World.OnLoadingFinished` (`Assets.Scripts.Objects.World`, static) -- **two Postfixes**, no args. Runs after `XmlSaveLoad.LoadWorld` finishes: all Things loaded, networks linked, `Thing.OnFinishedLoad` fired, `GameState == Running`. Server-only guard via `GameManager.RunSimulation` on each.
  - `ReplaceLongPiecesOnLoadPatch` -- rebuilds every placed long run from single-tile pieces (cable singles at `CableRoll.Canonical`).
  - `NormalizeCableRollOnLoadPatch` -- sweeps `GridController.AllStructuresPool` and re-rolls every `IsNormalisableStraight` cable to its run-axis canonical orientation. **Order-independent** with the long-piece rebuild: if the cable sweep runs first, the rebuild's freshly-created single-tile cables get canonicalised by `NormalizeCableRollOnRegisterPatch` when they register (the rebuild runs at `GameState == Running`); if it runs second, the rebuild already spawned its cable singles at `CableRoll.Canonical` so the sweep re-checks them and finds them already canonical. `IsNormalisableStraight` skips long cables and `IsBeingDestroyed`, so the sweep never touches a long run mid-conversion.

- `Cable.OnRegistered(Cell)` (`Assets.Scripts.Objects.Electrical.Cable`, override) -- **Postfix**, `(Cable __instance)`. `NormalizeCableRollOnRegisterPatch`: re-rolls a just-registered straight cable to canonical. Gated `GameManager.RunSimulation && GameManager.GameState != GameState.Loading`.
  - **Why `OnRegistered` and not a `Constructor.SpawnConstruct` prefix for the roll:** `OnRegistered` is the convergence point for *every* host-side fresh placement -- a plain click build, an R-rotated build, a ZoopMod drag, a BlueprintMod paste (`OnServer.Create<Thing>`, which does *not* go through `SpawnConstruct`), the remote-client-manual `ConstructionCreationMessage` path (also `OnServer.Create<Structure>` directly), and the single-tile cables this mod's long-piece rebuild and `RewriteLongVariantOnConstructPatch` spawn. A `SpawnConstruct` prefix would miss the BlueprintMod and remote-client-manual paths. The cost of using `OnRegistered`: by the time the postfix runs, `Thing.OnRegistered` has already snapshotted `RegisteredRotation` from the pre-canonical transform, so `Normalise` must also re-set `RegisteredRotation` (which it does). One harmless downside remains -- if the postfix runs before `SetStructureData` it can't on this path; `SetStructureData` runs first inside `SpawnConstruct`, but if it ever did run after, it would re-set `Structure.Direction` to the pre-canonical value. `Direction` is a network-sync-only field that no gameplay code reads (see "Game internals"), so a transient stale `Direction` is harmless and self-corrects on the next sync.
  - `CableRuptured` derives from `SmallGrid`, not `Cable`, so burnt/ruptured cables never hit this patch -- they are covered by the on-load sweep (`NormalizeCableRollOnLoadPatch` iterates every `Cable` instance and `CableRuptured` is not one). `Cable.OnRegistered`'s own body early-returns its `CableNetwork.Merge` step during `GameState == Loading`, so the postfix's `!= Loading` gate matches the game's own discriminator; `World.OnLoadingFinished` runs *after* `GameState` flips to `Running`, so the long-piece rebuild's `SpawnConstruct` output (and block 4's) registers with `GameState == Running` and does pass through this postfix.

- `Constructor.SpawnConstruct(CreateStructureInstance)` (`Assets.Scripts.Objects.Constructor`, static) -- **Prefix**, `(CreateStructureInstance instance, ref Structure __result)`. `RewriteLongVariantOnConstructPatch`: if `instance.Prefab` is a long variant, expands it into N single-tile `SpawnConstruct`s (cable bases at `CableRoll.Canonical(instance.LocalRotation)`, pipe/chute bases at the long piece's own rotation), sets `__result` to the first, returns false. If the long prefab's footprint is unavailable (null or `< 2` cells), or any exception is thrown, returns true (lets the long piece be placed; the on-load rebuild is the backstop). Early-returns `true` when `!Settings.MasterEnabled` (and naturally falls through when `LongToBase` is empty -- which is also the state when the master is off, or when every family toggle is off).
  - **Coverage:** `SpawnConstruct` is the host-side build chokepoint for vanilla manual builds, vanilla multi-kit builds (host and the remote-client multi-kit message handler `CreateStructureMessage.Process`, which re-runs `multiConstructor.Construct` host-side), ZoopMod (its commit replays into `MultiConstructor.Construct` -> `SpawnConstruct`), and `Cable.Break`'s rupture re-spawn. It does NOT catch a BlueprintMod paste (`OnServer.Create<Thing>` per entry, bypassing `SpawnConstruct`) -- a blueprint-pasted long piece is the one case still caught only by the on-load rebuild. With the long variants stripped from every kit's `Constructables`, ZoopMod's long-variant discovery returns empty and it places only single tiles, so this prefix is a no-op for ZoopMod-placed pieces; ZoopMod's `MultiConstructor.Construct` ignores `SpawnConstruct`'s return value, so even returning a single-tile `__result` for a hypothetical zooped long variant would be tolerated.
  - Recursion-safe: the inner `SpawnConstruct(new CreateStructureInstance(basePrefab, ...))` calls have `basePrefab` = the single-tile base, not a long variant, so they fall through the prefix. Uses `CreateStructureInstance`'s 5-arg ctor (same as the on-load rebuild), which forces `GridController.World` -- a non-issue for cables (no rocket-mounted long cables); a long pipe/chute would always be placed on the world grid anyway in practice.

- `SPDADataHandler.HandleThingPageOverrides()` (global namespace, `public void`, parameterless instance method; `Stationpedia.DataHandler` is a `static SPDADataHandler` field, called from `Stationpedia.PopulateLists` just before the per-prefab page build) -- **Postfix**, `(SPDADataHandler __instance)`. `HideLongVariantsStationpediaPatch`: re-asserts `HideInStationpedia = true` on every long variant in `LongVariantRegistry.LongVariants`, and re-writes `__instance.HiddenInPedia[longVariant.PrefabName] = true` (the dictionary `PopulateLists` also consults), so that if a future mod's XML page-override cleared the flag (`HandleThingPageOverrides` does `thing.HideInStationpedia = item.HideInSPDA` for each named prefab) the long variants are still skipped by the page build that follows. Gated `Settings.MasterEnabled`; a no-op when `LongVariants` is empty (vanilla ships no page-override for these prefabs, so the postfix normally re-hides nothing). Belt-and-suspenders -- the imperative `HideInStationpedia` write in `LongVariantRegistry.Build()` is the primary mechanism. See `Research/GameClasses/Stationpedia.md`.

Everything else this mod does is imperative (in `LongVariantRegistry.Build()`), not a Harmony patch:
  - `Thing.HideInStationpedia = true` on each long variant whose family toggle is on -> `Stationpedia.PopulateThingPages` skips it (page list, category lists, search). The field is normally written only from XML page-overrides; vanilla has none for these prefabs (a future mod's override is re-corrected by `HideLongVariantsStationpediaPatch` -- see the patch catalog).
  - `MultiConstructor.Constructables.RemoveAll(longVariant)` -> the kit mouse-wheel no longer offers it, and the `MultiMergeConstructor` merge code can no longer resolve to a long variant. Only long variants whose family toggle is on are in `LongToBase`, so a disabled family's kit is left fully intact.
  - `SmartRotate.AutomaticSetup(rotatable)` on the single-tile straight pipe/cable/chute base prefabs (every base whose long variants are stripped -- i.e. whose family toggle is on) plus -- only if `Settings.CableAlignmentEnabled` -- every straight cable of every tier, applied both to the registry prefabs and to the matching entries left in the kits' `Constructables`. The set is built once as `autoSetup`, a `HashSet<int>` of prefab hashes -- `LongToBase.Values` (the stripped bases: per-family-gated, so a subset of pipe / insulated pipe / liquid pipe / chute / super-heavy cable) union -- when cable alignment is on -- every `Cable` with `IsStraight` (normal, heavy, super-heavy -- the normal/heavy ones have no long variant but still want the `Straight` cursor; the super-heavy base is already in the first set when its toggle is on, so when cable alignment is off but the super-heavy toggle is on, the merge-with-tool fix still runs on it and the normal/heavy cables just don't get the cursor change). For a `(PlacementSnap.Grid, RotationAxis.All)` straight piece with two same-`NetworkType` open ends on one axis, `AutomaticSetup` switches `ConnectionType` from the C# default `Exhaustive` (24 box orientations) to `Straight` (3 orientations -- one per run axis with a fixed canonical roll) and sets `OpenEndsPermutation` to the matching axis triple (`{1,0,0,0,0,1}` / `{0,1,0,1,0,0}` / `{0,0,1,0,1,0}`); it touches nothing else (and falls back to `Exhaustive` -- a no-op for a piece that is already `Exhaustive` -- if it can't classify the geometry). Effects: (1) the build cursor's smart-rotate cycle (`C`-key / scroll) offers 3 orientations like a straight pipe instead of 24 (for cables this is a behaviour change; for pipes/chutes the vanilla `Straight`-typed long variants set the precedent and the single-tile bases now match it); (2) the cursor's default-pick lands on a canonical roll; (3) the merge-with-tool path (`MultiMergeConstructor.Construct`) iterates `Constructables` looking for an entry whose `GetConnectionType() == Straight` -- after the long straights are stripped (the only `Straight`-typed entries vanilla shipped in the pipe and cable kits), that entry only exists if the single-tile base is `Straight`, so this pass prevents a `NullReferenceException` / `KeyNotFoundException` in that path that the long-variant strip would otherwise introduce for pipe *and* cable kits (see "Pitfalls"). Chutes get the call too but `Chute` is not `IGridMergeable`, so they never hit that merge path -- the call is harmless there (a straight chute's build cursor just cycles 3 orientations like a pipe's). `AutomaticSetup` reads only serialized prefab data (`PlacementType`, `RotationAxis`, `AllowedRotations`, `GridSize`, the `OpenEnds` `Connection` list) and child-transform positions -- all present on a registered prefab at `OnPrefabsLoaded` time -- and is idempotent; it has zero other runtime callers in the game (editor-bake helper only). The roll-normalisation feature (`CableRoll`, the on-load sweep, the `Cable.OnRegistered` postfix) stays cable-only -- pipes/chutes don't visually glitch on a mixed roll, so only their build-cursor `ConnectionType` is touched, not their placed rotation. `LongVariantRegistry.LongToCellCount` is populated alongside `LongToBase` from the regex digit group -- diagnostic only (the log line "name says N segments"); block 4 reads the real footprint off `GridBounds`.
  - The long-variant prefabs are deliberately **not** unregistered from `Prefab.AllPrefabs` -- `XmlSaveLoad.LoadThing` does `Prefab.Find(thingData.PrefabName)`, so an old save containing a long piece would fail to load if the prefab were gone. They are left registered and the instances are deleted post-load instead.

## Game internals this mod relies on

(See `Research/GameClasses/MultiMergeConstructor.md`, `Constructor.md`, `CreateStructureInstance.md` for the full writeups.)

- `Prefab.AllPrefabs` (`List<Thing>`) -- every registered prefab. `Prefab.Find(string)` returns a `Thing` by prefab name. `Prefab.OnPrefabsLoaded` (`event Action`) fires once after `Prefab.LoadAll()` completes.
- `Structure.GridBounds.GetLocalSmallGrid(Vector3 position, Quaternion rotation)` returns `object` (`Grid3[]`): the small-grid cells the structure occupies, computed from the cached mesh-derived bounds (`Structure.CachePrefabBounds()` -> `GridBounds = new GridBounds(this)`). Valid on both prefabs (pass `Vector3.zero` / `Quaternion.identity`) and placed instances (pass world pos/rot). `Structure.BlockingGrids` is the post-placement registered set and is a single-cell placeholder on a fresh prefab -- do not use it for classification.
- `MultiConstructor.Constructables` (`List<Structure>`) -- the kit's option list, a serialized list of direct prefab references (not `Prefab.AllPrefabs` lookups). `MultiConstructor.OnPrefabLoad()` already null-strips it and clamps `LastSelectedIndex` before `OnPrefabsLoaded` fires. Index 0 is the single-tile base straight piece; ZoopMod depends on that, so removing the long variants is safe.
- `Stationpedia.Regenerate()` -> `Stationpedia.PopulateThingPages()` loops `Prefab.AllPrefabs` and `continue`s (no page) when `allPrefab.HideInStationpedia`. Not cached across runs; re-runs on language change; still skips because the flag persists on the prefab.
- `World.OnLoadingFinished(XmlSaveLoad.WorldData)` (`Assets.Scripts.Objects.World`, static, line ~305850) -- called at the tail of `XmlSaveLoad.LoadWorld`, after `World.OnLoadingFinished` -> `GameManager.StartGame()` would normally still need to run; actually it is `LoadWorld` that calls `OnLoadingFinished`, which calls `GameManager.StartGame()` synchronously (which flips `GameState` to `Running` and runs `UpdateThingsOnGameStart` -> per-Thing `OnFinishedLoad`). A postfix here therefore runs after all of that. There is no `WorldManager.OnFinishedLoad`; `GameManager.StartGame` is `async UniTask` (awkward to postfix directly), so `World.OnLoadingFinished` is the clean hook.
- `GridController.AllStructuresPool` (`Assets.Scripts.GridController`, `static readonly DensePool<Structure>`) -- the canonical "all placed structures" pool; every `Structure` is added in `GridController.Register` / removed in `Deregister`. Iterate with `ForEach`; snapshot before destroying.
- `OnServer.Destroy(Thing)` -- destroys the Unity object, which deregisters the structure's grid cells, removes it from its `PipeNetwork` / `CableNetwork` / `ChuteNetwork` (splitting the run into disconnected ends, queuing the gas redistribution as a `NetworkAtmosphereEvent`), and queues a `DestroyEvent` to clients. Server-only.
- `Constructor.SpawnConstruct(CreateStructureInstance)` (static) -- on the simulation owner does `Thing.Create<Structure>(prefab, worldPos, worldRot, 0L)` + `SetStructureData(localRotation, owner, localGrid, customColorIndex)`. `Thing.Create` registers the structure (which fires `OnRegistered` -> network merge, since `GameState == Running`) and queues it for replication to connected clients. Charges no item cost. On a client it just sends `ConstructionCreationMessage` and returns null (which is why this mod's loop is host-only). The vanilla `Cable.Break()` uses the same destroy-then-`SpawnConstruct` pattern.
- `CreateStructureInstance(Structure prefabToCreate, Grid3 localGrid, Quaternion worldRotation, ulong ownerClientId, int colorIndex = -1)` -- `localGrid` for a small-grid structure is the small-grid-snapped `Grid3` value `GetLocalSmallGrid` returns; `colorIndex = -1` means "use the prefab default colour"; `SetStructureData` only applies a color when `colorIndex >= 0 && colorIndex != CustomColor.Index`.
- `Thing.CustomColor` is a `ColorSwatch` (`.Normal` material, `.Index`). It is the `PaintableMaterial` swatch by default; a real paint override is `CustomColor.Normal != PaintableMaterial`. `Thing.HideInStationpedia` is a public `bool`.
- The `...Burnt` long cable variants are `CableRuptured` prefabs (electrically dead, no `CableNetwork`); their base is `StructureCableSuperHeavyStraightBurnt`. The name classification handles them without a type check; they need no network rejoin and carry no colour.

### Straight-cable roll degree of freedom (the cable-alignment feature)

The full game-internals writeup -- the placement-input -> rotation pipeline, the build-time creation chokepoints, the runtime-rotation-change recipe, the `SmartRotate` tables, `Structure.Direction` semantics, and the ZoopMod/BlueprintMod rotation derivations -- lives in the central knowledge base at `Research/GameSystems/PlacementOrientation.md` (sections "Straight pipe/cable/chute roll is an unconstrained cosmetic degree of freedom", "Changing a placed Structure's rotation at runtime", "Build-time structure-creation chokepoints", "Structure.Direction is reliably correct after every load path", "No gameplay code reads Structure.Direction", and the `SmartRotate` IndexOutOfRange section). What this mod relies on, in brief:

- **The roll is a real, cosmetic-only DOF for `IsStraight` cables.** `Cable.ConnectionType` defaults to `SmartRotate.ConnectionType.Exhaustive` (24 box orientations). For a "straight along axis A" cable, 4 of those 24 are connectivity-identical (the 0/90/180/270 deg rolls about A) but visually distinct, because the heavy / super-heavy cable mesh carries a band that breaks the axial symmetry. Connectivity / network membership / collision / the merge system / save-load are all roll-blind for a straight piece (`Connection.SetGrids` derives end cells from `position` and `position + forward*0.5`; rolling about the run axis -- which *is* `forward` -- moves neither end). Nothing in `SmartRotate` / `ISmartRotatable` normalises a placed cable's roll; the `OpenEndsPermutation` array is bookkeeping for the cursor's rotation-cycle, not a constraint on a placed piece. The small grid is 0.5 m (`SmallGrid.SmallGridSize`); a straight cable occupies one cell and that cell is rotation-invariant.
- **The cable mesh follows `transform.rotation` 1:1, no dynamic re-skin.** There is no `CableMesh` / `CableSkin` / `UpdateCableMesh` -- the visible cable is a static child renderer (unlike pipes, which dynamic-mesh). So changing a placed cable's `transform.rotation` at runtime makes the mesh follow the same frame, no refresh call. (This is also why pipes and chutes show the same roll inconsistency in their save data but rarely glitch visibly -- their meshes are axially symmetric and dynamic-meshed.)
- **`Cable.OnRegistered` is the JIT hook; `GameState != Loading` is the discriminator.** Every host-side fresh placement funnels into `Thing.Create<T>` -> `Referencable.RegisterNew` -> `Structure.OnAssignedReference` -> `GridController.World.Register` -> `Cable.OnRegistered(Cell)`. `Cable.OnRegistered` early-returns its `CableNetwork.Merge` body when `GameManager.GameState == GameState.Loading`, so a postfix gated `!= Loading` (and `RunSimulation`, to exclude joining clients whose `GameState` is `Joining`) cleanly excludes save-deserialise while still catching `World.OnLoadingFinished`'s post-`Running` rebuild output. `Thing.Create<T>(prefab, pos, rot, referenceId)` is reachable by both build and load paths; `referenceId == 0L` is the universal build-vs-load discriminator (every fresh build/spawn passes `0L`; load and join pass a non-zero saved/replicated id).
- **`Constructor.SpawnConstruct(CreateStructureInstance)` is the host-side build chokepoint** for vanilla manual + vanilla multi-kit (host and remote, the remote message handler re-runs `MultiConstructor.Construct` host-side) + ZoopMod. It is *not* on the save-load path or the join path, and BlueprintMod's paste (`OnServer.Create<Thing>` per entry) bypasses it. Mutating `instance.LocalRotation` in a prefix -- or returning false and looping N inner `SpawnConstruct`s -- is the clean place to rewrite a placement before `Object.Instantiate` runs (so `RegisteredRotation`, snapshotted synchronously inside `Thing.Create`, sees the rewritten value).
- **The verified-minimal runtime re-roll recipe** (server-authoritative): set `ThingTransform.rotation` (visible mesh follows) + `RegisteredRotation` (what the save's `StructureSaveData.RegisteredWorldRotation` and the join package's `Structure.WriteTransform` read) + `Direction` (the join-package `Structure.Direction` field, kept consistent) + on the server `NetworkUpdateFlags |= 1` (bit 1 = Thing transform delta, carried to connected clients on the next `SerializeDeltaState` tick). Do NOT re-register (`OnDeregistered`/`OnRegistered`) or call `RebuildGridState` -- that would re-run `Cable.OnRegistered` -> `CableNetwork.Merge` (a full network rebuild), and the single occupied cell is rotation-invariant so there is nothing to re-register. `RebuildGridState` is the "one engine call instead of three field writes" fallback but it still does not set the sync bit, so it saves nothing.
- **`Structure.Direction` is network-sync-only and reliably correct after every load path.** `Structure.OnAssignedReference` sets `Direction = ThingTransform.rotation` during `Thing.Create`, after `Object.Instantiate` has already applied the saved rotation; the late-join path additionally re-sets `Direction` explicitly in `Structure.DeserializeOnJoin`. No tick or gameplay code reads `Structure.Direction` (cable connectivity derives everything from open-end child transforms + `Connection.ConnectionType` bitmasks) -- it exists only to carry the placement quaternion across the late-join boundary -- so a briefly-stale `Direction` is harmless.
- **`SmartRotate.AutomaticSetup(ISmartRotatable)` only writes `ConnectionType` + `OpenEndsPermutation`** (and falls back to `Exhaustive` -- a no-op for a piece that is already `Exhaustive` -- if it can't classify the geometry). It never touches `AllowedRotations` / `RotationAxis` / `PlacementSnap`. For a `(Grid, All)` straight piece (cable, pipe, or chute) it produces `ConnectionType.Straight` + `OpenEndsPermutation` = one of `{1,0,0,0,0,1}` / `{0,1,0,1,0,0}` / `{0,0,1,0,1,0}` -- exactly the configuration the vanilla `Straight`-typed long variants (and almost certainly the straight pipes) ship with. This mod calls it on the single-tile straight pipe / cable / chute base prefabs (and the matching kit `Constructables` entries) plus every straight cable tier. It reads only serialized prefab data (`PlacementType`, `RotationAxis`, `AllowedRotations`, `GridSize`, the `OpenEnds` `Connection` list) and child-transform positions, all present on a registered prefab at `OnPrefabsLoaded` time (`Prefab.Register` -> `Thing.OnPrefabLoad` -> `CachePrefabBounds` has run), and it is idempotent. It has zero other runtime callers in `Assembly-CSharp` (editor-bake helper only).
- **The canonical roll set this mod re-rolls placed cables to** (v1.1-polish): one quaternion per run axis -- Z run `Quaternion.identity` (forward +Z, band/local-up on world +Y -- what a vanilla fresh-cursor Z straight comes out at), X run `Euler(0,90,0)` (forward +X, band on world +Y), Y run `Euler(270,0,90)` (forward +Y, band on world -X). The X and Y members are the per-axis plurality corner-mating optimum measured on a 12k-thing test save (`Research/GameSystems/PlacementOrientation.md`, "Why straight-cable roll normalisation does not fix the band-seam at a corner cable" -- the per-axis-plurality assignment there). (Earlier v1.1 used `RotationsList[SmartRotate.ConnectionType.Straight]` -- `Euler(0,0,90)` / `Euler(0,90,0)` / `Euler(90,0,0)` -- which turns out to mate with the *fewest* corners per axis; the v1.1-polish change moved it to the plurality optimum, which also fixes the reported 318512/318557 corner case.) The build-cursor `AutomaticSetup` pass still installs `ConnectionType.Straight` on the straight pipe/cable/chute prefabs (3-orientation cursor cycle) -- that is unchanged; it is the *placed-cable re-roll target* that this set defines. The mod classifies the run axis by the dominant absolute component of `rotation * Vector3.forward` -- placed rotations are always 90-deg box orientations, so this is exact.

## Multiplayer protocol

The world-mutating themes are host-authoritative; the settings have a small server-sync layer and a hard join-time enforcement.

- **Long-variant rebuild + cable-alignment on load:** the `World.OnLoadingFinished` postfixes run on the host / single-player only (`GameManager.RunSimulation`). `OnServer.Destroy` queues `DestroyEvent`s and `Constructor.SpawnConstruct` queues created structures, so the rebuilt world reaches connected clients through the standard server-to-client replication. A client loading a world through a host's save reload re-syncs the whole world anyway.
- **Cable-alignment on build:** the `Cable.OnRegistered` postfix runs only where `GameManager.RunSimulation`; `CableRoll.Normalise` raises `NetworkUpdateFlags |= 1` on the server, so already-connected clients get the corrected rotation via the next per-tick transform delta, and a late joiner gets it via the join package's transform (`Structure.WriteTransform` writes `RegisteredRotation`, which `Normalise` sets). The build cursor `AutomaticSetup` change is a prefab mutation each client applies locally on its own `OnPrefabsLoaded`.
- **Settings sync.** `MOD.Networking.RegisterMessage<SettingsConfigMessage>()` registers a host->client message carrying the eight settings bools (first seven slots stable; `RemoveLongInsulatedLiquidPipes` appended last so the wire read order stays stable). `SettingsConfigSync.HookHostBroadcast()` attaches a `SettingChanged` handler to each entry so a live host-side tweak `SendAll`s a fresh snapshot; the plugin's `IJoinSuffixSerializer` writes the same eight bools into the joiner's world-snapshot. On the client these only update a mirror used for logging -- they do NOT re-run the prefab-time effects (those happened at `OnPrefabsLoaded`, before the join) and don't need to: a value mismatch was already rejected at join time. The world rebuild + cable realign on load, and the build-time long-piece rewrite, are host-authoritative regardless.
- **Join-time enforcement.** `MOD.Networking.Required = true` makes LaunchPadBooster reject a joiner without the mod or with a different version (its own mod-version handshake; see `Research/Protocols/LaunchPadBoosterNetworking.md`). The plugin's `IJoinValidator` (`SerializeJoinValidate` / `ProcessJoinValidate`) adds the eight settings bools on top: both sides serialise their values in the same order (the seven stable slots, then `RemoveLongInsulatedLiquidPipes`), the read side rejects on the first differing setting with a readable per-setting error (`"setting 'X' does not match -- this side has it ..., the other side has it ..."`). It fires only on the remote-client-join exchange (`VerifyPlayer` / `VerifyPlayerRequest` postfixes inside LaunchPadBooster), not for single-player or a host loading its own world -- which is exactly right: the prefab-time effects on a host's own session use the host's own config, and there is no peer to disagree with.

Why the settings need the join-time enforcement and not just the sync message: the `Constructables` strip, the `HideInStationpedia` writes, and the build-cursor `AutomaticSetup` all happen at `Prefab.OnPrefabsLoaded`, which is *before* any client joins, so each machine first applies its own config. A client joining a host with different settings would see a different build-kit option list and Stationpedia than the host's world expects; the build-kit index a client clicks no longer lines up with the host's stripped list, and the server places the wrong piece. Pushing the host's settings after the join can't undo the already-loaded prefab state (a restart would be needed), so the clean fix is to reject the mismatch -- consistent with the version check.

## Pitfalls

- **Do not unregister the long-variant prefabs.** `XmlSaveLoad.LoadThing` resolves them by name; removing them from `Prefab.AllPrefabs` would silently drop every placed long piece in an old save before this mod can rebuild it. Keep them registered; delete the instances post-load.
- **Hook timing for the kit-strip.** `MultiConstructor.OnPrefabLoad` runs during prefab loading, before `Prefab.OnPrefabsLoaded`. Patching `OnPrefabLoad` and applying the patch from the `OnPrefabsLoaded` handler would miss the kit prefabs that already loaded. That is why the kit-strip is imperative (runs after, over `Prefab.AllPrefabs`), not a Harmony postfix.
- **Snapshot before destroying.** `GridController.AllStructuresPool` is mutated by `OnServer.Destroy` (via `Deregister`) and by `SpawnConstruct` (via `Register`). Collect the target list first, then iterate it. Same for the cable-alignment on-load sweep -- it snapshots the pool too (the long-piece rebuild running in the sibling postfix mutates it).
- **Index 0 of a kit's `Constructables` must stay the single-tile base.** ZoopMod assumes it. `RemoveAll` on the long variants preserves index 0 in practice; do not "tidy" the list further.
- **The `Burnt` regex case.** `^(Structure[A-Za-z]+Straight)(\d+)(Burnt)?$` matches `StructureCableSuperHeavyStraight10Burnt` (base = `StructureCableSuperHeavyStraightBurnt`) but not `StructureCableStraightBurnt` / `StructureCableStraightH` (no digit run before the suffix), and not the digit-less base pieces. Verified safe against the full `english.xml` prefab list; conveyors (`...StraightLong`) and rails (`...StraightStop`) do not match.
- **`SmartRotate.AutomaticSetup` must set BOTH `ConnectionType` and `OpenEndsPermutation`.** Flipping only `ConnectionType` from `Exhaustive` to `Straight` on a straight pipe/cable/chute prefab leaves `OpenEndsPermutation = {0,1,2,3,4,5}`, which is not a registered key in `OrientationLookup[Straight]` (whose keys are the 3 axis-rotations of `{1,0,0,0,0,1}`) -- so `SmartRotate._GetNext`'s `dictionary[openEndLocationPermutation]` throws `KeyNotFoundException` every cursor frame that kit is held. `AutomaticSetup` sets both, which is why this mod calls it rather than hand-flipping the enum.
- **Do not change a placed cable's roll via `RebuildGridState` or a deregister/re-register.** Both re-run `Cable.OnRegistered` -> `CableNetwork.Merge` (a full network rebuild) for what is a transform tweak. `CableRoll.Normalise` mutates fields on the live, already-registered cable instead; the occupied cell of a single-tile `SmallSingleGrid` is rotation-invariant so nothing needs re-registering. (`RebuildGridState` would also not set the client-sync bit, so even as a "convenience" call it saves nothing.)
- **`MultiMergeConstructor.Construct` for pipes and cables needs a `Straight`-typed `Constructables` entry once the long straights are stripped.** When you merge two collinear straight pieces with the kit's exit-tool in your off hand, `Construct` resolves the merged `connectionType = Straight` (the merge math keys off `OpenEndsPermutation`, not `ConnectionType`), then iterates `Constructables` for an entry whose `GetConnectionType() == Straight` -- and the only `Straight`-typed entries vanilla shipped in the pipe and cable kits were the long variants this mod removes. With them gone and the single-tile base still `Exhaustive` (the C# default), the loop finds nothing and the next line, `OrientationLookup[Straight][singleTile.GetOpenEndsPermutation()]`, throws (the single tile's `{0,1,2,3,4,5}` is not a `Straight` key). The merge-with-tool path applies to pipes (`Piping`, which is `IGridMergeable`) the same way it applies to cables, so this latent v1.0 regression hit pipe kits as well as cable kits. **v1.1 fixes it for both:** the `AutomaticSetup` pass runs on every single-tile base whose long variants are stripped (the pipe families, the chute, the super-heavy cable) plus every straight cable tier, so after `Build()` the single-tile straight *is* the `Straight`-typed `Constructables` entry and the merge degenerates to "snap orientation, consume nothing", error-free. Chutes get the same `AutomaticSetup` call but `Chute` is not `IGridMergeable`, so they never hit this merge path -- the call is harmless there (a straight chute's build cursor just cycles 3 orientations like a pipe's, which is consistent). The roll-normalisation half of the cable-alignment feature stays cable-only -- pipes/chutes don't visually glitch on a mixed roll, so only their build-cursor `ConnectionType` is changed, not their placed rotation.

## Design decisions

- **Replace, not delete, from v1.0.** The user's call. Plain `delete` would split the network at that cell. Rebuilding from single-tile pieces preserves the layout, rotation, colour and network *connectivity*. It does NOT preserve a pipe run's gas (see Pitfalls -> "Gas is not preserved across the rebuild") -- the destroy+respawn loses it and a capture+re-inject attempt didn't stick, so v1.0 ships with rebuilt pipe runs starting empty. It also loses any item physically in transit inside a long chute segment. Both are documented limitations; reliable gas preservation is on the TODO.
- **Settings panel (v1.1): master + per-family + cable-alignment toggles, server-authoritative, with join-time enforcement.** v1.0 shipped with no settings (the rationale at the time: the world-rebuild is host-authoritative, a per-player toggle would not undo the host's changes, and a server-synced toggle is more plumbing than a "these don't exist in this world" mod warrants). v1.1 adds them anyway: a master `Enable Network Purist Plus` toggle (off = the mod is inert: `LongVariantRegistry.Build()` short-circuits, every patch early-returns), six per-family long-piece toggles (`Remove Long Gas Pipes` / `... Liquid Pipes` / `... Insulated Gas Pipes` -- the insulated *gas* pipe only -- / `... Insulated Liquid Pipes` / `... Chutes` / `... Super-Heavy Cables`), and `Align Straight Cables`, all defaulting on, all with a `("RequireRestart", true)` tag (each gates `OnPrefabsLoaded`-time work, so a change needs a game restart). The pipe-family setting names spell out "Gas Pipe" / "Insulated Gas Pipe" even though the game's own term for the basic pipe is just "Pipe" (and "Insulated Pipe" is its insulated gas variant) -- a player reading the panel can't otherwise tell the gas toggle from the liquid one; the per-family classification (`LongPieceFamily` / `ClassifyFamily`) keys off prefab-name substrings, not these names, so it is unchanged. The per-family toggles gate everything for that family by being checked in `LongVariantRegistry.Build()` before the long variant is added to `LongToBase`; everything downstream (kit strip, `HideInStationpedia`, on-load rebuild, build-time rewrite, the family-base merge-with-tool `AutomaticSetup`) keys off `LongToBase`, so a disabled family is left exactly as vanilla. The cable-alignment toggle gates the on-load cable sweep, the `Cable.OnRegistered` re-roll, and the cursor `AutomaticSetup` of cable prefabs that have no long variant (`StructureCableStraight` / `StructureCableStraightH`); the family-base `AutomaticSetup` (the merge-with-tool fix for the stripped pipe/chute/super-heavy-cable bases) stays tied to the per-family toggles, not this one, because it is a correctness fix not a cosmetic one. **The settings are server-authoritative and enforced at join time, not merely synced**, because the prefab-time effects (the `Constructables` strip etc.) run at `OnPrefabsLoaded` -- before any join -- so each machine applies its own config; a mismatch desyncs the build-kit option lists and there is no way to fix that after the join short of a restart. A `SettingsConfigMessage` + `IJoinSuffixSerializer` push the host's values to the client for its own logging, but the `IJoinValidator` (alongside LaunchPadBooster's `Required` version handshake) is what actually keeps everyone consistent: a client whose mod-version or any of the eight settings differ from the host's is rejected on join with a clear message. This is the same join-handshake pattern PowerTransmitterPlus uses (`Research/Patterns/ConflictDetection.md`, `Research/Protocols/LaunchPadBoosterNetworking.md`).
- **Name predicate, not a hard-coded list.** Classifying long variants by "name matches `<registered straight>` + digits" catches mod-added long variants and the `CableRuptured` burnt variants without enumerating prefab names, and never misclassifies a single-tile base. (v1.0 also gated on a footprint check; v1.1 dropped it -- `GridBounds` turns out to be populated at `OnPrefabsLoaded` time so the check worked, but the name pattern alone suffices and the on-load rebuild reads the footprint off the placed instance regardless.)
- **No Harmony reject in `MultiMergeConstructor.Construct`.** ZoopMod drives placement through `InventoryManager.UsePrimaryComplete` with no per-piece error recovery; a mid-placement reject would produce half-built zoops. Emptying `Constructables` of the long variants makes ZoopMod fall back to single-tile placement cleanly, with no errors and no special-casing of ZoopMod.
- **Stationpedia handled via `HideInStationpedia`, not a page-removal patch -- plus a belt-and-suspenders re-hide (v1.1).** Setting `HideInStationpedia` is the same flag the game itself uses for hidden prefabs, set before the Stationpedia is built and resilient across language changes; a page-removal postfix would have to keep `StationpediaPages` and the private link lookup in sync, so the flag avoids that. The one way the flag can get cleared after `LongVariantRegistry.Build()` sets it: `SPDADataHandler.HandleThingPageOverrides()` (called from `Stationpedia.PopulateLists`, just before the per-prefab page build) walks the loaded XML `ThingOverrideData` and writes `thing.HideInStationpedia = item.HideInSPDA` for each named prefab -- so a future mod's page override with `HideInSPDA = false` would un-hide one of these. v1.1's `HideLongVariantsStationpediaPatch` is a Harmony postfix on `HandleThingPageOverrides` that re-asserts `HideInStationpedia = true` (and the parallel `SPDADataHandler.HiddenInPedia[prefabName] = true`, which `PopulateLists` also consults) on every long variant in `LongVariantRegistry.LongVariants` after that pass, so the page build that follows still skips them. Gated on the master toggle; a no-op when nothing was classified. (`SPDADataHandler` is a global-namespace type; `Stationpedia.DataHandler` is a `static SPDADataHandler` field. See `Research/GameClasses/Stationpedia.md`.)

### Cable alignment (v1.1)

- **Why all four building blocks (sweep + JIT-on-register + cursor `AutomaticSetup` + long-variant `SpawnConstruct` rewrite).** They cover disjoint cases: the on-load sweep fixes runs already baked into a save (the reported glitch -- 42% of straight-cable runs in the test save were roll-mixed); the `Cable.OnRegistered` postfix makes every *new* placement canonical the instant it registers (vanilla click/R, ZoopMod drag, BlueprintMod paste, and the single tiles the long-piece rebuild and block 4 spawn); the `AutomaticSetup` cursor pass makes the build preview / smart-rotate cycle / merge-with-tool path consistent (3 orientations like a pipe, and -- not optional -- it stops the merge-with-tool path from throwing now that the long straights are stripped from `Constructables`); the `Constructor.SpawnConstruct` long-variant rewrite catches a long piece placed mid-game (vanilla manual/multi-kit, ZoopMod) immediately instead of waiting for the next world load (this was already on the v1.0 TODO). No single block covers all of those; together they make the mod's "this network is tidy" promise hold at every point a cable can enter or change. The cursor-`AutomaticSetup` pass is the one of the four that is *not* cable-scoped: it runs on the single-tile straight pipe and chute base prefabs too (every base whose long variants are stripped), because the merge-with-tool `KeyNotFoundException` it prevents is a property of the merge path, which pipes (`Piping`, `IGridMergeable`) hit just like cables -- so the same v1.0 regression existed for pipe kits. Chutes get the call harmlessly (`Chute` is not `IGridMergeable`). The other three blocks -- and the roll-normalisation behaviour generally -- stay cable-only: pipes and chutes don't visually glitch on a mixed roll, so there is nothing to normalise about their *rotation*; only their build-cursor `ConnectionType` is set to `Straight`.
- **Why this canonical set specifically (v1.1-polish).** The set is: Z run `Quaternion.identity` (band on world +Y), X run `Euler(0,90,0)` (band on world +Y), Y run `Euler(270,0,90)` (band on world -X). The Z member is the orientation a vanilla fresh-cursor Z straight comes out at (no R presses), so a click-built Z cable is already canonical. The X and Y members are the *per-axis plurality* corner-mating choice: of the (up to four) connectivity-equivalent rolls per run axis, this is the one whose band face the largest single share of corner cables on that axis exposes their band to (measured on a 12k-thing test save -- X-axis corner-legs: band-on-+Y 43%, the largest of the four; Y-axis: band-on--X 32%; Z-axis: band-on-+Y 31%; see `Research/GameSystems/PlacementOrientation.md`). It is *not* `RotationsList[SmartRotate.ConnectionType.Straight]` (`Euler(0,0,90)` / `Euler(0,90,0)` / `Euler(90,0,0)`), which earlier v1.1 used -- that set's Z roll puts the band on world -X (the Z-axis *second* choice) and its Y roll on world +Z (the Y-axis second choice), so it mates with the *fewest* corners per axis; the v1.1-polish change swapped to the plurality optimum, which strictly dominates the old choice on whole-corners-matched (~36% both legs vs ~26%) and fixes the reported 318512/318557 case. **This does not make every corner mate** -- a corner's band-exit face is per-corner-rotation (not even a function of its leg-direction pair: the "L flipped over" cases), the per-axis demand is roughly a 40/40/10/10 split over four world faces, and a straight between two corners with contradictory band faces can match at one end only, so the best any one-roll-per-axis assignment achieves is ~36% of corners fully matched. The build-cursor `ConnectionType.Straight` `AutomaticSetup` pass is unchanged (separate concern -- it gives the cursor a 3-orientation cycle and prevents the merge-with-tool `KeyNotFoundException`; it does not constrain a placed cable's roll). Future work for an actual corner fix: re-roll each corner-adjacent straight to its specific corner's band face (with a tie-break for a straight between two disagreeing corners; possibly normalise corners too) -- see `TODO.md`.
- **Applies to all cable tiers, gated `IsStraight`.** Normal (`StructureCableStraight`), heavy (`StructureCableStraightH`), super-heavy (`StructureCableSuperHeavyStraight`) -- every `Cable` with `IsStraight == true` that isn't a long variant. Never corner / tee / cross / junction cables: their open ends sit off the roll axis, so rolling them *would* move open ends and break connectivity. The `IsNormalisableStraight` predicate enforces this; the long-variant exclusion (`!LongToBase.ContainsKey`) keeps the alignment paths out of the way of the long-piece-rebuild paths.
- **Cosmetic-only, host-authoritative, no functional change.** A straight cable's roll affects nothing gameplay reads -- it's the visible mesh orientation and nothing else. The v1.0 caveats (rebuilt pipe runs lose gas; rebuilt chutes lose items in transit; multiplayer-with-client-during-host-save-load not exercised in detail) are unchanged by v1.1 -- the cable-alignment work neither touches gas/items nor adds any new client-state path beyond the existing `NetworkUpdateFlags` transform delta and the standard server-to-client replication the long-piece rebuild already relies on.

## Risks not fully resolved by research (carried into v1.0)

- **`...Burnt` long cable prefab class.** The decompiled assembly only shows `Cable.RupturedPrefab : CableRuptured`, strongly implying `StructureCableSuperHeavyStraight10Burnt` is a `CableRuptured` prefab; the prefab class itself is in the asset YAML, not the DLL. The mod handles burnt long cables the same way (replace with N x `StructureCableSuperHeavyStraightBurnt`) via the type-agnostic predicate; if a burnt variant turns out to behave unexpectedly, the per-piece try/catch logs and skips it. Burnt long cables are rare in practice (you have to electrocute a long cable).
- **Pipe volume bookkeeping.** A long pipe's serialized `volume` is expected to be roughly N x the single-tile pipe's volume, so the network's total volume is unchanged after the rebuild. If a long variant's `volume` is not proportional, the network's steady-state pressure shifts slightly after the rebuild. Not verifiable from the DLL (asset data). Acceptable.
- **Gas is not preserved across the rebuild (v1.0).** `OnServer.Destroy` is deferred to end-of-frame; the resulting `Pipe.OnDestroy` copies the *whole* network gas and queues an async `NetworkAtmosphereEvent.DivideNetworkAtmosphere` to the surviving connected networks. When a pipe network's pipes are wholesale replaced this way the gas does not arrive at the rebuilt single-tile network -- field-confirmed: rebuilding a gas-edited test run (4 mol O2 + 16 mol N2 + 123 kJ, 190 L) left the rebuilt 18-cell run with `Oxygen=0 Nitrogen=0 Energy=0`. A fix that captured each affected `PipeNetwork.Atmosphere.GasMixture` (via `GasMixtureHelper.Create().Set(...)`) before destroying and re-injected it (`Atmosphere.Add(...)`) into the rebuilt network after a 3-frame settle was tried; the re-inject *ran* (logged "restored gas to N rebuilt pipe network(s)") but the gas still ended at zero in the save -- could not pin down whether the captured `GasMixture` struct didn't copy, `Atmosphere.Add` didn't take, or a late `DivideNetworkAtmosphere` overwrote it, without runtime-instrumenting. v1.0 therefore does not claim to preserve gas; the rebuilt pipes start empty. Re-attempting needs a deeper dig into `NetworkAtmosphereEvent` / `AtmosphericsManager` ordering (and probably forcibly *setting*, not adding, the rebuilt network's atmosphere after the events fully settle, with diagnostics that log the captured/before/after mole counts).
- **Multiplayer with a client connected during a host save-load.** Loading a save on the host is a full world reload; clients re-sync. `OnServer.Destroy` queues `DestroyEvent`s and `SpawnConstruct` queues created structures for replication, so the rebuilt world reaches connected clients through normal sync. Not exercised in detail; treat as a thing to watch in multiplayer testing.
