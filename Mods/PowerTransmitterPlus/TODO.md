# PowerTransmitterPlus TODO

- [ ] **Verify `Structure.Direction` reconciliation post-save-load on a non-floor dish.** Per `Research/GameSystems/PlacementOrientation.md` Open Questions, the `Direction` field on Structure (the per-instance backing Quaternion that gets shipped via `SerializeOnJoin` / `DeserializeOnJoin`) is NOT explicitly restored from `StructureSaveData`. It IS reassigned from `ThingTransform.rotation` in `OnAssignedReference` and `RebuildGridState`, both of which fire during grid registration on the load path; in practice this should keep `Direction` aligned with the placed orientation. Confirm empirically with a wall-mounted dish placed on host, save, reload, snapshot `(Structure.Direction, ThingTransform.rotation)` via InspectorPlus and check they match. If a gap is found, the fix is a save-data extension or an `OnFinishedLoad` postfix that copies `ThingTransform.rotation` into `Direction`.

- [ ] **Tune the `OnRegistered` H/V default for non-floor placements.** `PowerTransmitter.OnRegistered` and `PowerReceiver.OnRegistered` reset to `(H=0, V=1)` on fresh placement (skipped at save load). V=1 corresponds to the dish-local `+Y` axis: world-up for a floor mount, world-down for a ceiling mount, and world-outward for a wall mount. Defaults are sensible (always away from the mount surface) but if player feedback says wall-mounted dishes should default to a horizon-aligned direction instead of straight out, override the default H/V via a Postfix on those `OnRegistered` overrides that branches on the structure's mount-orientation.

- [ ] **Filter `_constructionCursors` lookup to actual game-loaded prefabs.** The current `PlacementPatcher.MutateExistingCursors` walks the entire `_constructionCursors` dictionary and matches by type. Cheap (one type-check per cursor) but visits every cursor in the dictionary on every host launch. If a future profiling pass shows this is hot, replace the loop with a direct-by-name lookup once the prefab names are confirmed via UnityPy.
